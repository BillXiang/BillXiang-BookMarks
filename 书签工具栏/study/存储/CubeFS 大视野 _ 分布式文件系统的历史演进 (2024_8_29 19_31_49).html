前言



文件系统是支持文件的数据持久化、访问，并支持以目录形式进行组织管理的存储系统。


文件系统是一个老古董，长期以来维持着基本的概念未变。例如文件文件系统最基础的概念是 inode 和 dentry，从 60 年代到现在的主流单机文件系统、分布式文件系统，无论是哪个领域或者形态，都在使用这两种基础的文件系统的字段结构。这是因为他们在分别负责文件的查找遍历（dentry）和元数据信息存储（inode）两个维度，只有这样配合使得在查找和访问文件时保持高效，进而提升整体性能和响应速度。但同时，各个文件系统的 inode 和 dentry 内的字段定义又不完全相同，除了保留最进本元数据信息，更多还是为了适配不同的存储的布局、查询的方式、特性能力的实现。


文件系统也是一个不断革新进步的产品，支撑计算机领域的发展和突破。


不变的是对于性能、成本的要求，高效的组织文件，提供高效的文件读写查询。


变化的是对系统能力的要求的提升，支持海量文件存储，校验能力、故障恢复能力，容灾能力，针对用户各种不同特性需求如回收站、统计、审计、生命周期、快照、克隆等。


变化的还有应用场景，例如公有云场景、AI 场景，例如存算分离场景、跨区域复制的场景，同时希望系统有更好的自动化运营能力等等。



单机文件系统历史演进



演进过程


文件系统（File System）是操作系统用于管理和存储文件的技术，随着计算机技术的发展，文件系统也经历了多次演进。从早期简单的文件系统到现代复杂的分布式文件系统，每种文件系统都有其独特的设计和应用场景。


计算机发明初期并没有文件系统。


60 年代开始 IBM 的计算系统开始包含文件系统，标志着文件系统开始成为操作系统的一部分。


70 年代的 Unix 初创，就伴随了自己的文件系统，也形成了后期 POSIX 标准。


80 年代开始的 FAT 系列，包括（FAT12, FAT16, FAT32），这个时期及以前，空间需求都相对有限，例如 FAT16 支持 2GB，而且实现简单，也在后期嵌入系统的手持设备经常被用到。


90 年代的 windows 的 NTFS，开始支持了现代文件系统的高级特性，例如文件压缩、加密、细粒度权限控制、事务日志等，当然复杂性也逐步提高。


同一时期，linux 的 EXT 系列也开始逐步演进，并以稳健著称，被广泛采用。


2000 年前后，随着磁盘空间的增加，大容量磁盘，大分区、机柜等开始成为主流需求，EXT4、XFS 等开始普及。


随后，一些新的文件系统出现，比较虽然 Btrfs、ZFS、F2FS 和 ReiserFS 等文件系统在某些场景下有其特点，但它们的影响力和广泛使用程度相较于 EXT4 和 XFS 较小。


虚拟文件系统


Linux的文件系统


Linux 采用了虚拟文件系统（VFS）概念，通过在内核中创建一个软件抽象层，将物理存储介质上的文件系统与用户之间连接起来。VFS 通过创建一个通用的文件模型，兼容所有类型的文件系统。这使得 Linux 能够支持大多数现有的文件系统。


下图是 linux 的 inode 和 detnry 的定义。dentry 通过文件名与 inode 关联，当访问文件时，系统首先查找 dentry，然后通过 dentry 获取对应的 inode 以访问文件的元数据和数据块。



1、inode 结构体包含了文件的元数据和操作信息。

部分字段如：


  • 文件的访问时间、修改时间

  • 数据的具体位置

    对于本地系统一般是磁盘偏移数组

  • 对于分布式系统位置信息要能路由到目标，可以是自己的维护的路由定位方式，也可以是外部系统提供唯一标识的数组


2、dentry 结构体记录了文件名称相关，当用户输入路径时，实际是从根目录一层一层找到对应的文件。所以 dentry 就是用于文件名查找的关键。


  • dentry 关联了其对应的 inode 信息，这样可以找到 inode。dentry 相比 inode 内容更少，较少的信息量，更适合缓存，用于目录的遍历查找。

  • inode 和 dentry 这两个结构体共同支持了 Linux 文件系统的操作和管理。文件 dentry 和目录 dentry 可以共用结构，和文件系统自身设计相关。


这个通用模型主要基于 Linux 的 EXT 系列文件系统构建。每个特定的文件系统需要将其物理结构映射到这个通用模型中。例如，在通用模型中，所有目录被视为文件，这些文件可以包含其他文件和目录。而在某些文件系统（如 FAT）中，目录并不被视为文件。为了适应通用模型，内核会在内存中生成相应的目录文件。此外，在执行实际文件操作时，VFS 会通过指针调用特定文件的操作函数。



Windows 的文件系统


Windows 也有类似于 VFS 的抽象层。这个层次的设计允许 Windows 支持多种文件系统，并提供统一的接口来访问这些文件系统。


Windows 使用文件系统驱动程序来实现与不同文件系统的交互，允许操作系统和应用程序以一致的方式访问文件。这些驱动程序提供了标准的 API，使得文件操作（如打开、读取和写入文件）可以跨不同文件系统进行。通过这种抽象层，Windows 能够支持多种文件系统，如 NTFS、FAT32、exFAT 和 ReFS，使用户和应用程序不需要关心底层实现。



分布式文件系统的历史演进



共享文件系统


共享文件系统也是分布式文件系统核心的特性之一，换言之，共享文件存储相当于分布式文件系统种子。


大学实验室孕育了早期的文件共享技术，较为有代表的是 NFS，1989 年 NFS 发布了 v2 版本，NFSv2 在许多大型企业和大学中获得应用，支持多台计算机和用户同时访问共享文件系统。随着其在网络文件共享中的成功，NFSv2 的使用规模逐渐扩大，成为跨平台文件共享的标准之一。


同一时期，还有 SMB，不过 SMB2.0 得到广泛认可是在 2006 年。


一个是 linux，一个 windows，也代表了用户的需求和行业趋势。



分布式存储系统


随着互联网普及，头部互联网厂商需要海量数据的存储，从而诞生了分布式文件系统，如 GFS、HDFS。


GFS


GFS（Google File System）是在 2000 年代初期由 Google 开发的，旨在满足其大规模数据处理的需求。当时的背景如下，也是互联网普及初期，头部互联网厂商普遍的需求。


1、数据增长，随着互联网的迅猛发展，面临着海量数据的存储和处理挑战。


2、传统文件系统的局限，传统文件系统在处理大文件和高并发访问时存在性能瓶颈。


3、分布式计算需求，需要一个能够支持分布式计算框架（如 MapReduce）的大型分布式文件系统，以实现高效的数据处理。


4、容错性和可扩展性，为了确保系统的高可用性和对硬件故障的容忍，能够通过添加新节点来扩展存储能力。


GFS 是开创性的产品，打破了人们对于数据存储的传统认识，推动了数据存储和数据处理的发展，具有里程碑的意义。



论文 《The Google File System( https://research.google.com/archive/gfs-sosp2003.pdf)》 成为了后来者 HDFS 的核心设计思路。图上的 ChunkServer 概念是 GFS 首创，在以后的各种存储系统上依然沿用至今。


HDFS


HDFS（Hadoop Distributed File System）可以视为 GFS 的开源实现。HDFS 与现有的分布式文件系统有许多相似之处。但是，与其他分布式文件系统的区别也很大，HDFS 参照 GFS 论文，放宽了一些 POSIX 要求，以实现对文件系统数据的流式访问。HDFS 具有高度的容错能力，旨在部署在低成本硬件上。HDFS 提供对应用程序数据的高吞吐量访问，适用于具有大型数据集的应用程序。



HDFS 如今依然是大数据领域非常基础的存储平台，因为其相对简单的架构设计，活跃的社区支持，不断演进的能力，使得 HDFS依然保持着举足轻重的地位。


分布式文件系统


发展历史


2010 年前后，云的概念开始在头部厂商内部落地，各大厂商（国外如 AWS、Microsoft、Google、国内 BAT）开始完善自己的海量存储系统，并基于底层系统构建不同的接入服务能力，从而来适配各自强劲的业务需求。


2015 年，移动互联网经过了几年的高速发展，热潮达到了顶峰，海量存储和业务需求又到了一个新的阶段，集群成本的控制，性能、成本、稳定性、易维护、容灾调度能力等都又了新的需求，同时对中台建设形成了共识，各大厂商开始构建自己的中台，对外提供了相对完善的云计算服务。


这一时期对于存储系统的需求更为具体和明确，其中通用性成为了一个重要特点，例如对于 POSIX 文件的操作修改等标准接口、能力需求又回归到了构建一套分布式文件存储系统本身，人们需要一套系统解决海量文件存储的各种问题，其中 Ceph 逐步成为影响力比较大的分布式文件存储系统。


CephFS 是基于 Ceph 的对象存储实现的分布式文件系统，它利用 Ceph 的底层对象存储来管理文件和目录，提供高可用性和可扩展性。



但基于对象的 CephFS 分布式文件存储，初期架构元数据系统 MDS 不可扩展，再加上百万行级别的代码的系统维护极其困难，一般情况下需要一个专业的团队来运营 CEPH，而且因为MDS的亲和性和锁粒度问题，很容导致死锁问题，影响线上运营。从而，这一时期，大厂都开始对标 CephFS 研发自己的文件系统。


CubeFS


CubeFS 作为新一代元原生存储产品，具备简洁的架构设计、可扩展的元数据服务和优异的读写性能。该产品在SIGMOD发表论文《CFS: A Distributed File System for Large Scale Container Platforms》中首次亮相，并于2019年对外开源；同年，CubeFS 加入 CNCF 开源基金会，成为其“沙箱”项目，2022年进入了“孵化”阶段。目前，CubeFS 已经在 OPPO、京东、贝壳、网易、小米、BIGO 等互联网公司中大规模的生产应用。

分布式文件存储对比其他存储产品



通常会遇到一些业务，不知道如何选择底层存储，或者对分布式文件存储也存在一定的误解。


和本地文件系统的区别


一、性能差异


1、网络延迟


在分布式文件系统中，数据通常需要通过网络传输，这会引入额外的延迟，即使是在高速局域网中也是如此。网络延迟会影响数据的读写速度，尤其是在高频率的随机读写操作中更为明显。这就直接意味着对 iops 的影响，尤其是要求 direct=1 的直接持久化。但对于流式的写入，因为是异步的响应，优化得当是可以忽略延迟影响的。


2、通信开销


分布式文件系统需要处理节点间的通信，包括数据复制、一致性保证以及故障恢复机制等，这些都需要额外的通信开销。这些通信开销增加了数据传输的复杂度和时间消耗。同时，client 和服务端、服务端之间，网络处理也会消耗较多的 CPU。这种类似于 direct=1 的影响，但会影响系统的整体吞吐能力。


二、数据差异


1、元数据管理


单机文件系统，系统级元数据就是超级块（SuperBlock），包含了描述整个文件系统的重要信息，如文件系统的类型、块大小、可用和已用的块数量、inode 表的位置等。不同的文件系统有不同的超级块格式和布局。文件系统的不同位置保存超级块的备份，以增强数据的可靠性。对于文件系统则以其他形式存在。


分布式文件系统发展到现在，主流都是集群模式支持多租户的形式，超级块即集群维度及卷维度的元数据信息可以单独存储，例如存储到 rocksdb 自己做主从或者 etcd；也可以基于自己的存储系统存储，实现系统自举，目前 CubeFS 是单独存储的，单独部署可以避免和存储系统的网络分区问题，类似于管控面和数据面分离。分布式文件系统中的元数据（例如文件属性、权限、位置信息等）的操作，如查找、更新等，都需要通过网络进行，这也会增加延迟。


2、数据复制和冗余


为了提高数据的可靠性和可用性，分布式文件系统通常会将数据复制到多个节点上。数据的复制过程会占用网络带宽，并且增加写入延迟。


本地文件系统一般不使用 raid 卡做冗余，即便使用也没有网络消耗、分区导致的通讯问题，连带需要异步修复等问题。


三、性能与均衡


1、并发控制


分布式文件系统需要处理多个客户端同时访问同一份数据的情况，这就需要复杂的并发控制机制。并发控制机制可能会导致性能瓶颈，尤其是在高并发场景下。


2、负载均衡

分布式文件系统需要不断地监控节点的状态，并动态地分配数据和任务以保持负载均衡。负载均衡算法和实施会带来额外的计算和网络开销。


四、故障处理、恢复、容灾


分布式文件系统必须设计成能够容忍节点故障，多分存储的一致性在故障期间得到保障。


故障不可恢复情况下要支持数据和元数据的迁移，迁移机制、自动化迁移能力对系统要求极高，也非常复杂。


故障检测和恢复机制会增加系统的复杂性和响应时间。


容灾能力增强，除了多副本的容灾，也要支持多 AZ 部署，单 AZ 异常，不影响系统读写。


五、服务特性支持


平台提供的冷热分层、回收站、流控、审计等附加能力。


六、成本差异


本地存储空间会存在很多的浪费，对于分布式文件存储平台，会以多组户的形式，对于存储资源共享。同时可以有效的利用高密度地磁盘，降低运营成本。



和块存储的区别


虽然块存储上面可以格式化文件系统，但两者有明显的区别和适应场景。


1、分布式文件系统适用于需要文件共享和协作的应用场景。


2、块适合单机的存算分离，但不需要共享的场景。


3、块存储接口只有读写删或者高级的特性，不要考虑上层文件系统类型和其内部的元数据存储和管理，对于文件系统的元数据和数据是统一处理。


4、块存储适无法理解文件系统或者其他系统的指令，多端无法实施感知和控制文件系统元数据，从而无法控制并发写入造成的数据冲突。也无法控制用户文件系统的变化，导致共享读异常。


5、分布式文件系统更加适配海量文件的元数据及相关特性的支持，块存储更适合单机存算分离的场景。


和对象存储的区别


1、层次结构


文件存储采用类似本地计算机或网络驱动器的层级目录结构。文件被组织成目录和子目录。


对象存储使用平面命名空间，不采用层级目录结构。对象通过键来唯一标识，并存储在桶或容器中。


2、协议


文件存储支持 POSIX 接口，也就适配了通过 NFS（网络文件系统）、SMB（服务器消息块）或 FTP（文件传输协议）等协议访问。


对象存储通常通过 RESTful API 或 HTTP(S) 协议访问。当然 CubeFS 是兼容的。


3、修改写能力


文件存储支持修改写，对象存储需要用户自己维护修改。很多基于对象存储的文件系统就是自己实现，采用 COW（Copy-On-Write）和 ROW（Rewrite-On-Write）的实现方式。


4、传统定义的适应场景


文件存储最适合需要层级目录结构和文件级别访问的应用程序。通常用于传统的文件共享和需要将文件组织在目录中的应用程序。


对象存储适用于处理大量非结构化数据的应用程序，如多媒体内容、备份、档案和大数据分析。它提供高度可扩展和耐用的存储解决方案。



和其他存储产品之间的转换



通过文件系统支持对象存储


CubeFS 就是通过文件系统的原型来支持对象存储，可以参考社区文章《 CubeFS 产品核心能力剖析 | 一文带你深入理解 CubeFS 对象存储 》。


CubeFS 对象存储基于文件系统的底层逻辑进行设计，在兼容资源管理子系统、元数据子系统、数据子系统的原始语义的基础上，实现文件到对象的互相映射。



通过文件系统支持块存储


块在客户端一般两种形式，一个是内核驱动，一种是用户态驱动。


不管那种输入，用户请求最终都是转换为网络请求到后端存储服务，后端存储服务提供数据的读写。很多块存储的后端服务不直接管理裸盘，而是使用了单机存储系统的文件系统，使用了 punch hole 的能力，CubeFS 也是如此。这里通过一些变化调整，是可以实现块存储的。


方式一：把单机替换为分布式文件系统，把多个存储节点收敛为一个存储节点，自然可以实现基于文件系统的块存储。进一步，取消掉存储节点，在块存储的客户端侧调用分布式文件系统的 sdk，类似于块客户端直接使用了一个大的文件系统。


但这样的问题在于，块存储上层还需要一个文件系统，多重文件系统，转换效率很低。


方式二：后端服务提供针对块存储的定制服务，对于块存储不需要提供元数据服务，只有磁盘空间的管理和编排划分，基于集群空间管理，提供块的管理。基于现有 CubeFS 的数据分区 DataPartiton 维度进行块维度的空间划分，取消掉细粒度的对于文件存储的数据分片管理。在客户端实现一套独立的基于用户态驱动的块客户端。


总的来说，基于方式二，CubeFS 系统的数据子模块经过裁剪、适配来实现对于块存储的支持是可以考虑的，如果朝着这个方向改造，相遇于一个存储引擎同时支持了文件、对象和块存储的方式。


通过对象存储支持文件存储


以 CephFS 为例，Ceph 的基础是 RADOS（Reliable Autonomic Distributed Object Store），提供高可用性和高性能的对象存储。CephFS 是 Ceph 的文件系统接口，提供 POSIX 兼容的文件存储。CephFS 将文件系统的元数据和数据分别存储，利用 RADOS 的高可用性和扩展性。


数据存储：CephFS 的数据部分以对象的形式存储在 RADOS 中，利用 RADOS 的高可用性和扩展性。


元数据存储：元数据（如文件名、目录结构等）也存储在 RADOS 中，但由专门的元数据服务器（MDS，Metadata Server）管理。MDS 负责处理文件系统的元数据操作，提高了元数据的访问效率。


基于 RADOS 实现的文件系统（如 CephFS）在性能上相较于直接写入磁盘管理确实会受到一些影响，主要体现在以下几个方面：


  • 额外的抽象层：RADOS 提供了对象存储的抽象，增加了一层处理，使得数据和元数据的访问相较于直接磁盘写入略显复杂。

  • 网络延迟：RADOS 在分布式环境中操作，数据可能需要通过网络进行传输，这会引入一定的延迟。

  • 元数据服务器（MDS）：元数据的管理由 MDS 负责，虽然 MDS 优化了元数据的访问，但在高并发情况下，可能成为性能瓶颈。

  • 数据处理时间：数据在写入之前需要进行分块、校验和其他处理，相比直接写入磁盘的操作，处理时间会更长。



CubeFS 的产品优势



1、具备完整能力的自闭环的存储产品


核心模块从客户端、管控、两个数据存储子系统、元数据系统等所有模块都是社区自己开发维护的，从而所有的代码都能够有很好的维护支持，团队能够更为系统的规划未来演进，定制业务需求，不需要担心第三方组件带来的需求匹配问题。


2、高性能元数据管理


  • 可扩展性：元数据分片，新建文件 inode 随机分布，元数据节点可增加。

  • 强一致：基于 raft 实现元数据副本的一致性。

  • 弱亲和性：目录 inode 和子文件 dentry 的关联关系，父目录 inode 和子文件 inode 没有绑定关系。

  • 基于 Btree 的元数据内存索引提供高效查询性能

  • 基于 rocksdb 的持久化索引（ 稳定性测试中 ），适配成本优化，这里后续可以借助 rocksdb 实现更多的元数据分片操作和 inode、dentry 的亲和性调整。


3、多协议兼容


CubeFS 支持了 HDFS 接口，结合大数据纠删码，支持数据入湖，作为存储底座，把海量、成本、性能等都一站式解决。


CubeFS 支持了协议兼容，文件和对象可以互通访问，业务可以根据需要选择或者一起使用。对象存储因为接口是 restful 的形式，这样对于业务不需要额外的配置或者库就能够使用。


CubeFS 提供 SDK 开发接口，对于存算分离，减少 Fuse 等组件对于性能的影响，从而可以更好的提高性能。


4、混合云管理


一期：CubeFS 混合云存储一期（ 稳定性测试中 ）


  • 基于多存储介质类型的数据空间划分、管理规则、可控机制

  • 基于多存储介质类型的数据的生命周期模型、迁移规则、迁移机制

  • 元数据索引对接基于多存储介质类型，配合迁移实现无缝的索引变更


基于一期我们可以实现基于生命周期规则的数据的单向迁移，例如冷热分层，从 SSD 副本到 HDD 副本存储，或者 SSD 副本到纠删码存储。


二期：实际上，基于混合云一期构建的基础，还可以实现


  • 数据分层流动，可以为后续提供基于生命周期的数据的流动规则，不仅仅是单项流动。根据规则配置可以实现多种的数据迁移能力。不仅仅是数据中心内的流动，也可以是云上云下的迁移调度。

  • 支持第三方存储，可外接 S3 等其他存储，提高更为丰富的数据部署能力。

  • 数据备份，根据生命周期规则，实现数据自动备份，构建备份恢复规则，元数据索引支持创建新的对应备份元数据信息。

  • 数据碎片整理，和本地文件系统一样，碎片产生原因较多也比较复杂，碎片整理有助于系统的稳定性和减轻空间的管理负担。可以通过生命周期来实现。

  • 数据分区收缩，数据分区管理单元过多对系统管理有一定的负担。通过混合云可以搬迁数据分区逻辑单元，从而注销掉过多的数据分区。


5、多层次的读写加速能力


加速能力是 CubeFS 近期投入的重点，也会是 CubeFS 后面比较突出的一点。加速也可以算作是混合云的一部分，这个对于云上云下数据的同步尤为重要，目前主要分几个方面：


  • 内部的读写性能的优化

    基于 Fuse 的一些并发能力受限考虑放开

    内部的关键路径继续优化，减少性能损耗

    写入流程优化，减少 meta 路径的频繁请求


  • 全链路 RDMA 能力改造，RDMA 是未来的标配场景，读写路径、同步路径都会带来显著的收益。


  • 分布式缓存，尤其对于AI场景，云上的算力使用有很大帮助。我们会基于此点在云上构建更多的能力。



小结



新的业务场景催生新的存储需求，推动分布式文件系统从小到大、从简单到强大的持续发展；新一代技术浪潮风起云涌，给CubeFS产品带来全新的机遇与挑战，CubeFS将砥砺前行，在性能、功能、成本、稳定性等核心产品能力上持续打磨，成为云原生AI开源存储领域的领导者。

CubeFS 简介




CubeFS 于2019年开源并在 SIGMOD 发表工业界论文，目前是云原生计算基金会 (CNCF) 托管的孵化阶段开源项目。作为新一代云原生分布式存储平台，兼容 S3、POSIX、HDFS 等协议，支持多副本和纠删码引擎，提供多租户，多 AZ 部署、跨区域复制等特性；适用于大数据、AI、容器平台、数据库及中间件存算分离，数据共享、数据保护等广泛场景。
