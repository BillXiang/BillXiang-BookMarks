Linux HMM（Heterogeneous Memory Management，异构内存管理）是一个内核子系统，旨在支持设备和主机共享和管理内存。这对于需要高性能计算的应用程序尤为重要，如 GPU 加速的应用、深度学习模型训练等。HMM 提供了一种机制，使设备（例如 GPU）能够直接访问主机内存，从而实现更高效的内存利用和数据传输。

HMM提供的4个功能：

  • 共享虚拟内存（Shared Virtual Memory, SVM）：

    • 允许主机和设备共享同一地址空间，这样应用程序可以更方便地在主机和设备之间传递数据，而不需要显式的数据拷贝。

  • 页面迁移（Page Migration）：

    • 当设备需要访问的数据不在它的内存中时，内核可以将页面从主机内存迁移到设备内存，反之亦然。这个过程对于确保数据的一致性和有效访问至关重要。

  • 页面失效（Page Fault）处理：

    • 当主机或设备访问不在其内存中的页面时，会触发页面失效中断。内核负责处理这些中断并将页面迁移到正确的位置。

  • 同步机制：

    • 确保主机和设备之间的数据一致性。内核需要提供机制来同步主机和设备对共享内存的访问。

支持SVM功能的代码示例

以下是一个使用 HMM 的简单示例，展示如何初始化和使用共享内存：

// 请求一块共享内存
    char *addr = mmap(NULL, SIZE, PROT_READ | PROT_WRITE, MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
    if (addr == MAP_FAILED) {
        perror("mmap");
        close(fd);
        return EXIT_FAILURE;
    }

    // 写入数据到共享内存
    strcpy(addr, "Hello, HMM!");

    // 假设此处有设备对共享内存的访问
    // LaunchKernel(..., addr, ...)

    // 读取共享内存中的数据
    printf("Shared memory contains: %s\n", addr);

HMM工作原理

下面根据上面代码的执行流程，简单讲述HMM的工作原理。HMM扩展 ZONE_DEVICE，添加 ->page_fault() 和 ->page_free() 事件回调函数，以允许设备驱动程序协调与设备内存（通常是 GPU 内存）相关的内存管理事件。

驱动初始化

GPU驱动在初始化时需要将设备内存通过devm_memremap_pages将设备内存注册为ZONE_DEVICE系统内存。注册时会传入一个叫migrate_to_ram的callback。这个函数负责在CPU缺页中断时将设备内存迁移到系统。

开辟系统内存

第一步malloc或者mmap分配内存addr，在这个过程中和设备驱动是没关系的，就是一个普通的系统内存分配。

设备访问addr

  • 第二步LaunchKernel后，GPU在运算时会访问addr，GPU遍历自己的页表发现找不到这个地址，因此触发GPU缺页中断。这是一个系统内存地址，GPU页表找不到这个地址是正常的。

  • GPU驱动会处理缺页中断（具体可以参考 https://github.com/torvalds/linux/blob/master/drivers/gpu/drm/amd/amdgpu/gmc_v10_0.c#L99）。在GPU缺页中断要做的事：

    • 页面迁移，从系统内存迁移到GPU（https://github.com/torvalds/linux/blob/master/drivers/gpu/drm/amd/amdkfd/kfd_svm.c#L2886） GPU驱动会根据addr建立自己管理的prange，放入到svm中，这样系统内存和GPU内存在此建立了关联。另外驱动会通过mmu_interval_notifier_insert_locked建立和系统MMU的hook，系统在更新页表时会及时通知到GPU。
    • 更新GPU页表 （https://github.com/torvalds/linux/blob/master/drivers/gpu/drm/amd/amdgpu/amdgpu_vm.c#L950）

这样addr的内存被迁移到了GPU，系统页表会将addr标记为换出。

CPU访问addr

因为addr已经被迁移到GPU，CPU访问addr时会触发缺页中断。在处理缺页中断时会发现页面被换出，系统会调用回调函数migrate_to_mem将内存从GPU迁移回CPU。

amdgpu page fault之后不会去创建页表，只是简单的报错。


amdgpu的SVM页表更新是在：

svm_range_validate_and_map->svm_range_map_to_gpus->svm_range_map_to_gpu->amdgpu_vm_update_range之后通过SDMA更新GPU页表。

这需要amdgpu里的kfd的maptogpu的ioctl接口，设备如果不通过显式的接口去访问系统内存，而是直接由硬件获取物理页表很容易会产生安全问题，属于非法访问，所以个人认为gpu page fault之后直接报错是正确行为。
