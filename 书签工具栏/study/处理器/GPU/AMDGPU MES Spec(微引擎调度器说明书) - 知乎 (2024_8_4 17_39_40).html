微引擎调度器（MES）固件负责在AMD GPU上调度图形和计算任务。本文档通过描述关键的调度器固件（MES）和硬件（队列管理器）组件，概述了AMD GPU的调度架构。

调度需求

从高层次来看，调度要求可以概括为：

  • 公平且高效地调度应用程序在GPU上的工作
  • 针对各种用户场景实现多个优先级别

这些高层次要求也可以从用户场景的角度进行描述：

  • 相同优先级的应用程序应该获得相等的GPU执行时间
  • 具有用户焦点的应用程序（例如，合成器）应该获得更多的GPU时间，但不能无限制地饿死正常优先级的任务
  • 实时任务如VR应立即运行，并且可以无限制地饿死低优先级的任务
  • 低优先级的任务如OneDrive、照片增强、压缩或Folding@home仅应在所有更高优先级的任务空闲时运行

调度器通过四级队列优先级实现上述要求。

这种调度行为与微软的GPU调度规范相一致。这些要求已在微软的GPU调度规范中详细说明，此处不再赘述。

硬件架构

调度器固件的主要作用是将调度需求映射到硬件架构上。因此，需要了解硬件架构，才能理解调度固件如何在AMD GPU上实现调度需求。

下图描述了调度/运行应用程序队列的高层次硬件架构和执行流程。

硬件架构的关键要点可以概括如下：

  • GPU前端有三个微处理器，分别用于执行调度、计算和图形固件。
  • 有多个图形（GFX）和计算（Compute）管道，每个管道包含一个队列管理器（Queue Manager），该管理器仲裁连接到该管道的一定数量的硬件队列。
  • 调度有两个层级：
    • 第一层调度在固件中进行，固件决定哪些应用程序队列应映射到各个管道上的可用硬件队列。
    • 第二层调度在队列管理器硬件中进行，队列管理器选择一个准备好的硬件队列并在着色器复杂体上运行。尽管第二层调度由队列管理器硬件完成，调度器固件仍然可以通过硬件队列优先级、时间片等各种调节参数来影响队列管理器的硬件队列选择和执行。
  • 队列管理器的仲裁逻辑选择一个硬件队列并在着色器复杂体上运行。被选中执行的映射硬件队列称为“连接队列”。
  • 每个管道提供了一条独立路径，用于在3D/CS复杂体内启动队列的工作。因此，理论上可能有多个“连接队列”并行运行。
  • 图形和计算工作共享一个ALU池。

调度器固件架构

调度器固件架构可以分解为以下关键组件：

  1. 调度器 API

这些是由驱动程序发送的命令，用于通知调度器有关事件的信息，如队列创建、销毁、挂起或其优先级的任何变化。每个 API 都在后面的 API 部分进行了描述。

  1. 调度器上下文

数据结构，调度器在其中维护应用程序、队列状态或任何其他调度状态或配置。调度器上下文是 API 处理器和核心调度线程工作的状态。调度器上下文包括：

  • HW资源状态

    • HQD状态 – 当前映射的队列，队列类型，计划时间。
    • VMID状态 - 当前映射的进程。
    • GDS状态 – 当前使用GDS分区的进程。

  • 进程调度状态

    • 调度级别状态 - 进程列表，宽限期，正常带百分比，是否有就绪队列（has_ready_queues）。
    • 进程状态 – 每个上下文优先级（-7/+7）的组列表，进程时间片，运行时间累积。
    • 每组状态 – 队列列表，运行时间累积，组时间片。
  1. API处理器

处理来自驱动的命令，并在需要时修改调度器的各种状态信息。

  1. 核心调度器

根据状态执行具体的调度动作。

  1. 中断处理

处理各个模块产生的中断。例如，中断处理程序从获取器中读取API数据，或收集各种硬件队列的繁忙、空闲状态。

调度算法

在描述调度算法之前，这里有一些队列术语及其描述，以帮助理解队列状态转换：

  • 用户队列 表示来自应用程序的绘制或分发命令的线性流。它在CPU世界中类似于一个线程。用户队列被分配了少量的内存资源，如环形缓冲区，其中应用程序提交命令包，以及一个内存来保存队列被抢占时的硬件执行状态。用户队列本身不执行。它需要映射到硬件队列上才能执行。

  • 硬件队列 一个硬件描述符，保存用户队列的状态（例如环形缓冲区地址、读写指针等）。硬件队列可能处于已映射或未映射状态。而已映射的队列可能处于连接或断开状态。

  • 队列映射/取消映射 映射是将用户队列状态加载到硬件队列上的行为。而取消映射是将队列状态从硬件队列描述符移动到内存中的行为。只有在抢占之后，硬件队列才能被取消映射。

  • 已连接队列 由队列管理器选择以在3d/CS复合体上运行的硬件队列。

队列可能的转换状态如下图：

基于这个说明，队列可能处于以下状态之一：

  • 未映射 用户队列尚未初始化到硬件队列中，仅存在于内存中。

  • 已映射且断开连接 用户队列已被初始化到硬件队列中，但目前未连接到着色器子系统，因此无法执行。

  • 已映射且已连接 用户队列已被初始化到硬件队列中，并已连接到着色器子系统。只有已连接的队列才能请求并在着色器资源上启动其工作。只有有待处理工作的队列才允许连接。

GPUSCH的实现可以分为两个步骤来解释：首先，我们进入轮询调度；其次，我们查看如何实现不同级别的队列优先级。

Round Robin调度

轮询调度（Round Robin Scheduling）是指基本的轮询调度算法，其中来自所有应用程序的队列具有相同的优先级，调度器应为每个应用程序提供等量的GPU时间。

调度器通过以下方式实现这一点：

  • 维护一个包含来自所有应用程序的队列的数据库。
  • 根据它们的调度顺序将它们映射到可用的硬件队列上。上述提到的数据库是调度上下文，它包含每种唯一队列类型（图形、计算、DMA）和优先级级别的队列列表。

在调度器上下文中总共维护了12个队列列表。

调度器上下文还包含队列或进程特定信息，如MQD指针、VMID或分配给队列或进程的任何特殊资源。来自驱动程序的各种API会导致调度器上下文中的队列和进程信息得到更新。

调度器随后会根据对调度器上下文的任何更新执行某些调度操作，如队列映射或取消映射。

AMD GPU具有固定数量的管线，每个管线都有一组固定的硬件队列。用户队列必须映射到硬件队列上才能执行其工作。由于硬件队列的数量有限，调度器将尝试尽可能多地将用户队列映射到硬件队列上。

当用户队列被映射到硬件队列上时，调度器会配置一个该队列必须运行的量子时间。一旦量子时间到期，队列管理器将在同一管线上连接下一个硬件队列。

当硬件队列未被过度订阅（即用户队列数小于等于硬件队列数）时，调度器会将所有用户队列映射到硬件队列上，并为所有队列配置相等的量子时间。

为了确保有限数量的硬件队列得到尽可能最佳的使用，调度器仅将包含待执行工作的用户队列进行映射。这要求调度器在未映射的队列中有新工作时得到通知。

这是通过使用聚合门铃（Aggregated Doorbells）来实现的。聚合门铃是特殊的门铃，当软件向未映射的队列提交工作时，会写入这些门铃。写入聚合门铃会通知调度器未映射的队列中有新工作。然后，调度器根据该队列相对于其他工作的优先级，尽快将该队列进行映射。当聚合门铃不可用或被软件使用时，调度器会开始定期轮询未映射队列的写指针内存，以发现它们是否有新工作。这仅在队列订阅过多，即所有用户队列无法映射到有限的硬件队列上时才会进行。

队列优先级

调度器尽可能多地将用户队列映射到可用的硬件（HW）队列上。一旦硬件队列被过度订阅，调度器就会开始将用户队列以轮询方式映射到可用的硬件队列上。

然而，当涉及到执行不同优先级级别的工作时，这种基本的轮询方案就显得不足了。调度器使用各种硬件优先级特性的组合来实现每个优先级级别所需的调度行为。

在讨论调度器如何使用这些优先级特性之前，了解调度器可用的各种硬件优先级特性是很有用的：

  • 中间命令缓冲区抢占 队列抢占是在各种优先级场景中实现所需服务质量的最基本特性。抢占可以在几个不同的工作边界上发出，这会影响延迟和需要保存或恢复的状态量。例如，计算工作可以在提交、分发、线程组或在着色器指令边界上被抢占。抢占延迟和保存或恢复的状态量将根据抢占粒度而有所不同。

  • 波限制 此方法通过限制可以发出的波的数量来减少其他队列的工作量。“波”代表一组着色器线程。

  • 管线优先级 每个管线上的已连接队列会向着色器硬件断言一个管线优先级。着色器硬件使用此优先级来选择和启动即将到来的工作，这些工作基于管线优先级。

  • 分发隧道 当从高优先级队列执行分发时，该方法会立即禁用来自其他队列的工作。分发隧道的能力被配置为队列属性。

  • 队列量子 量子由队列管理器硬件和调度器固件共同实现。队列管理器根据调度器固件在硬件队列中配置的量子来连接和断开队列。当队列超额订阅时，调度器会断开时间片耗尽的队列连接，选择其他用户队列。

  • 队列连接优先级 调度器根据队列优先级选择连接哪个用户队列。

  • 计算单元保留 这种方法允许划分出一定数量的计算单元，并仅供特定队列使用。这种方法用于机器利用率启动延迟至关重要的场景中。
