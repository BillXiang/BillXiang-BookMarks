intel 新的cpu架构sapphire rapid，支持了用户态接收中断：

  1. 用户进程发送中断给用户进程
  2. kernel发送中断给用户进程
  3. 设备发送中断给用户进程

其中第三点可以使得设备bypass kernel直接与用户态处理程序进行通信，dpdk之类的hack似乎可以不必要了。

注：支持嵌套中断

用户进程间发送中断

目前是SENDUIPI这个指令，格式如下：

senduipi reg // reg中是一个中断索引编号

用户态中断的基础设施

UITT: user interrupt target table，存放一个中断索引到中断接收方的映射表，建立中断发送方和中断接收方的连接，使得中断可以被投递。

UPID: user posted interrupt descriptor，中断描述符，描述了中断的信息：中断的接收cpu、vector（一个中断接收方以有多个中断号接收）：

UIPI的硬件处理流程

在这两个结构的基础上，执行SENDUIPI指令后，硬件先将跟进中断index从UITT表中找到中断接收方的UPID，然后对UPID进行中断标记设置（UPID.PIR）。

在此之后，一个IPI中断被硬件发出到目标CPU（UPID.Notification destination)，中断向量为（UPID.Notification vector）

目标CPU收到IPI中断，应答ACPI中断，检查中断向量的值，如果值是UINTR_NOTIFICATION_VECTOR，走用户态中断的处理流程。这一步称为用户态中断识别。

否则是一个普通的IPI中断，按照普通中断处理方式走。

进入用户态中断处理：

从UPID.PIR中读取中断向量（设置为1的所有未），然后设置到用户态中断寄存器（UIRR）中。用户态代码执行时硬件检查中断寄存器，触发用户态中断处理函数的执行。

Kernel对用户态中断的支持

GitHub - intel/uintr-linux-kernel

[RFC PATCH 00/13] x86 User Interrupts support

目前的状态是，支持sendUIPI的支持，也就是用户态到用户态的中断发送。

一系列syscall辅助构建硬件工作需要的数据结构：

uintr_register_handler ：注册中断函数
uintr_register_sender：注册发送函数，返回一个发送的index
uintr_create_fd：对一个中断编号返回一个fd，供后续使用
uintr_wait：挂起等待中断

使用场景

用户态发送IPI，为啥不用共享内存呢？感觉意义不大。

如果是为了唤醒等待方，传统的方法是调用futex_wakeup的syscall，这里使用UIPI的确可以减少发送者进入内核态的开销，但是对于接收方来说都是从内核态醒来。

io-uring的作者说可以用在替代uring的eventfd上，貌似没想到怎么有收益的。
