VM HW dirty log track

Inochi Amaoto
  • All Messages By This Member
9:26am   #667  

Since I have not finished the detail proposed of the ISA, This brief
is sent for the discussion.

== Introduction

The dirty log tracking is necessary for VM live snapshot, migration and
profiling. Currently, the dirty log tracking is implemented by using
write-protected page, which is somewhat ineffective and CPU costly.
So hardware acceleration for the dirty log tracking is necessary.

== Dirty Log Buffer

For hardware, the dirty log buffer is introducted to reflects the change
of the guest memory space. The dirty log buffer is used to record limited
history to avoid the overhead of processing guest page table.

The dirty log buffer is designed as a non-circular FIFO. Its size can
can decided at runtime and vary from 4KB to 2MiB to adapt different
realtime and granularity requirements.

Only effective guest store operations are record. An effective guest
store operation is a commited guest store operation without any error,
and it can be explicit or implicit

The recorded guest store operations are inserted at the writer index
of the dirty log buffer, then the index is incremented. If there is
already an entry with the same guest physical address in the dirty log
buffer, the hardware can ignore the new duplicated one. When the dirty
log buffer is full, an interrupt will be triggered and cause the guest
VM exit.

The address of the dirty log buffer will be set through an CSR interface,
and the software can access the last entries index through an CSR
interface. The order of entries in the dirty log buffer may not reflect
the real order of the guest store operations.

To reduce the complexity, all entries of the dirty log buffer should
be handled together. The hardware will always fill at the start of the
dirty log buffer after the buffer is reclaimed.

The format of the entry in the dirty buffer is as follows. The length
of entry is always SXLEN. For RV32, only lower 32 bit is used.

|63 54|53 10|9 0|
---------------------------------------------------------------
| Reserved | PPN | Reserved |
---------------------------------------------------------------

== CSRs

There are two new CSRs to hold the information of the buffer and state

1. Hypervisor Guest Dirty Trace Queue (hgdtq)

This CSR is HRW, XLEN bits, for RV32, only low 32 bits is used.
[0] : INH : Inhibit the dirty track
[1:4] : SIZE : the size of track buffer, this field represent the
buffer size is 2 << (SIZE + 12)
[10:53] : PPN : the physical page frame number of the buffer

2. Hypervisor Guest Dirty Trace Queue Index (hgdtqi)

This CSR is HRWC, 32 bits
[0: 32] : INDEX : The last index of filled entries in the buffer.
The start index is 1. A write will set this field
as 0

== interrupt

A new "Dirty Log Buffer Full" interrupt is introducted to handle the
dirty log buffer full event.

== possible enhanced points

1. Ranged tracking (Track memory change based on PTE)

This draft only allows to track all guest memory change when enabled.
In addition to profiling, I do not find a user case for this. So I
do not add support for tracking dirty page based on the PTE.

2. Different tracking granularity

This draft only does the track in a page size granularity. The entry
may represent an large size, or just contain the PTE of the guest page
table for processing.

3. Allow partial processing of dirty log buffer

To reduce CSR usage, only one CSR is used to represent the index of
last filled entry in the buffer, which prevents partial processing
of the buffer. I am not sure whether it is necessary. But I guess it
may not take too much time to process the whole buffer.

4. Access tracking

This is the same as 1, I think tracking guest memory access is only
for profiling.
