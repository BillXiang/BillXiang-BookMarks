简介

为了追求更高的性能，加速器持续发展，与 GPU、网络和存储的集成将更加紧密，并且整个系统中将存在大量非易失性内存 (NVM)。在下一代计算系统中，众多异构设备如何高效工作使得我们必须要思考。充分发挥这些系统的潜力需要系统软件的重大创新。不幸的是，现代操作系统 (OS) 虚拟内存管理系统没有高层次的内核编程接口 (KPI)，无法让外围设备轻松利用现有的虚拟内存 (VM) 机制。异构系统不再只管理 CPU 上运行的进程的地址映射。相反，它现在还必须管理虚拟化、I/O 设备、加速器、GPU、NVM 和系统其他组件的地址空间。迄今为止，这些不同的外围设备一直由独立的、不连接的系统管理，这导致操作系统中虚拟地址空间管理、物理内存管理和地址映射系统的激增。

为每个设备实现和维护独立的内存管理系统代价高昂。随着每个新设备添加到系统中，设备和操作系统开发人员一直在“重新发明轮子”，为设备构建独特的地址空间管理器、内存分配器和映射管理器。这些系统通常针对设备的独特特性进行优化，但仍然完成相同的基本任务。而且，不可避免地，它们会缺少核心 VM 系统中存在的特性或性能优化，这主要是由于复制核心 VM 系统中随时间发展而变得复杂的机制的难度太大。

独立的外围设备内存管理也带来了新的协调挑战。外围设备可以具有独特的页表格式和独特的同步机制，所有这些都必须同时支持。当地址空间在 CPU 和一个或多个设备之间共享（或仅在设备之间共享）时，这些不同的页表必须保持一致。I/O 设备还具有比 CPU 更高的地址映射更新率，以维护安全性和与设备的隔离。此外，许多外围设备都有自己的本地物理内存。该内存必须在整个系统范围内进行管理，VM 系统必须确定是否要在设备内存和主机内存之间迁移数据，或者是否要强制使用远程访问。最终，这些挑战使一些看似简单的操作系统任务变得极其复杂。例如，如果操作系统想要销毁所有对内存中特定物理页面的映射，那么现在它可能必须了解所有可能持有该页面的映射的子系统，并以特定于设备的方式销毁每个映射。最先进的技术是使用通知程序来简化此过程。例如，Linux 的 MMU 通知程序机制可用于通知二级 VM 系统，核心 VM 系统已使共享页面无效。虽然这种机制促进了二级 VM 系统与核心 VM 系统之间的协调，但它并没有消除对设备驱动程序中二级 VM 系统的需要。同时，操作系统和设备驱动程序之间引入了许多代码维护问题。

本文提出了用于外围设备的通用内存管理 (GMEM)。GMEM 为利用异构内存资源的系统提供集中式内存管理。具体来说，GMEM 引入了三个关键的创新。首先，GMEM 在操作系统中提供了一个用于跨所有设备管理内存的高级别接口。这使设备能够有效地利用操作系统的内存管理子系统。每个设备驱动程序只需要实现必要的设备特定代码，使操作系统能够与硬件交互，而不是为每个设备发明一个全新的内存管理系统。其次，GMEM 在设备之间利用 VM 系统中的功能和优化。因此，随着操作系统集成额外的功能和性能优化，基于 GMEM 的设备驱动程序会自动受益。最后，GMEM 允许在 CPU 和设备之间使用单个地址空间。这提高了加速器的可编程性，因为 GMEM 的内部协调机制执行必要的内存管理任务，使程序员无需担心地址空间协调。通过几个案例研究证明了 GMEM 系统的有效性。例如，将 FreeBSD 的 IOMMU 驱动程序转换为使用 GMEM 的 KPI 可以消除对特殊用途 IOVA 分配器的需要。这消除了六百多行代码，并降低了性能错误的可能性。此外，随着 GMEM 集成单独的内存优化，基于 GMEM 的 IOMMU 驱动程序会自动获得性能提升。实验表明，基于 GMEM 的驱动程序比主线 Linux 获得高 54% 的网络接收吞吐量，同时 CPU 利用率降低 32%。同时，基于 GMEM 的 IOMMU 驱动程序以比 Linux 低 45% 的 CPU 利用率实现了线路速率传输吞吐量。另一个例子是 Intel GPU 驱动程序被修改为使用 GMEM。GMEM 集成了一个内存优化，可以加快更大的页面清零速度，这将 GPU 内核启动时间减少了 45%。GMEM 还可以简化驱动程序开发。两个模拟 GPU 的驱动程序只需要不到 70 行代码来实现其设备内存管理，不包括内存管理单元 (MMU) 功能。可以在单个地址空间内快速开发用于这些模拟设备的应用程序，并且它们可以与较小的设备内存容量无缝执行。这些案例研究表明，GMEM 可以显著减少驱动程序代码大小、提高性能并简化驱动程序开发。

背景

虚拟内存管理

操作系统（OS）的 VM 系统将程序中使用的地址与实际物理内存分离。它是借助硬件 MMU（内存管理单元）来实现的。为了提供虚拟内存抽象，虚拟内存系统执行四个关键操作：虚拟地址管理、逻辑地址映射、物理地址映射和物理内存管理。

虚拟地址管理。当新进程启动时，VM 系统为该进程创建一个新的虚拟地址（VA）空间并设置该初始 VA 空间。要分配额外的地址或更改现有分配，程序必须进行系统调用（例如 mmap()、munmap()、sbrk()、exec() 等）。这些系统调用会分配、更改或取消分配虚拟地址，然后程序可以使用这些地址。其中一些系统调用还可能包括对这些地址范围的内存访问权限的声明或更改。请注意，分配虚拟地址时实际上不会分配物理内存。

逻辑地址映射。如前所述，当系统调用分配虚拟地址范围时，实际上不会分配物理内存。相反，创建一个逻辑映射来定义将物理内存分配给该虚拟地址范围时初始值将来自何处。例如，物理内存可以用文件内容或零填充。

物理地址映射。VM 系统提供虚拟到物理映射管理来控制 CPU 的 MMU 的地址转换过程。页表存储从虚拟地址到物理地址的现有有效映射。硬件 MMU 使用页表来翻译地址并将这些翻译缓存到转换旁路缓冲器 (TLB) 中。如果页表中没有有效的翻译（或访问者没有使用翻译的适当权限），硬件将导致页面故障，然后 VM 系统可以处理它。

当分配虚拟地址时，不会立即创建物理映射。相反，物理映射是在首次访问时创建的。当程序访问尚未物理映射的已分配虚拟地址时，页面故障处理程序将创建虚拟到物理地址映射。如果已经分配并准备了物理内存来支持该虚拟地址，则可以立即创建映射并将其安装到页表中。否则，必须首先分配并准备物理内存（将在下文讨论）。当进程恢复时，硬件 MMU 将重新尝试进行翻译，然后可以访问物理内存。请注意，如果进程访问了未分配的虚拟地址或没有访问类型的相应权限，页面故障处理程序将终止程序。

映射是在页面大小粒度上创建的。X86-64 处理器支持 4KB、2MB 和 1GB 页面大小。VM 系统负责选择页面大小，并可能根据各种事件（例如 Linux 中的“透明大页面”或 FreeBSD 中的“自动超级页面提升”）更改页面大小。当虚拟地址被取消分配时，其关联的映射也必须被销毁。这意味着从页面表中删除映射，并使缓存任何硬件 MMU 中的映射无效。

物理内存管理。VM 系统不会在分配的虚拟地址被访问并需要物理映射到新物理内存之前分配物理内存。当页面错误发生时，如果虚拟地址有效并且尚未分配物理内存，VM 系统将在创建虚拟到物理地址映射之前分配物理内存。物理内存可以一次分配单个页面（使用任何有效的页面大小），或者可以一次分配多个连续的页面。

分配物理内存后，必须对其进行“准备”。这可能涉及将页面归零（如果页面尚未清除）、复制内存（例如，支持写时复制）或从磁盘上的文件中读取内容。支持该物理内存的虚拟地址的逻辑映射指示了如何准备物理内存。

如果没有空闲的物理内存，则 VM 系统必须“回收”物理内存以供使用。为了回收页面物理内存，可能首先需要将其存储到磁盘，以便以后在下次访问时可以恢复它。如果页面内容已经存在磁盘上或否则不需要，则可以立即回收页面。

设备内存管理

历史上，设备可以直接、不受限制地访问机器的内存，而内存完全由受信任的内核代码控制。此外，设备内的任何固件都是受信任的。但是，今天许多设备由不可信的第三方驱动程序控制，这些驱动程序可能会导致安全问题。因此，外围设备用户可以使用 VA 空间隔离其内存资源，以限制来自其他设备的访问。表 1 将这些用例分为四种示例，具体取决于 VA 空间是否与主机进程共享以及设备是否具有自己的本地物理内存。

BUS_DMA 接口支持每个设备的私有 VA 空间，以隔离每个设备只能在其自己的 VA 空间内访问允许的机器内存。同样，Nvidia 的 CUDA 编程模型支持每个 GPU 的私有 VA 空间，以保护存储在 GPU 本地物理内存中的数据。还有一些支持共享 VA 空间的努力，包括 OpenCL 的共享虚拟内存 (SVM) 和 Nvidia 的统一虚拟内存 (UVM) 系统。例如，UVM 统一了 GPU 和 CPU 的地址空间，因此 UVM 程序无需管理设备缓冲区或跨 VA 空间协调传输。KVM 直通设备在访客虚拟机和设备之间共享 VA 空间和物理内存。本文提出的GMEM 系统为设备驱动程序提供了一个通用的高级接口来支持表 1 中示例的所有可能用例。相比之下，现有的操作系统支持非常有限，如下面详细说明。

之前的工作没有解决设备私有虚拟地址的问题。FreeBSD 和 Linux 的 BUS_DMA 接口都实现了二进制搜索树来管理私有 VA 空间中的 I/O 虚拟地址。核心操作系统 VM 系统已经为管理 CPU 上的进程地址空间实现了这样一个系统，但是设备驱动程序无法轻易地重用这些核心系统。这不是一个孤立的问题。事实上，大多数外围设备的内存管理任务与 CPU 的任务并无不同。但是，核心 VM 系统没有暴露高级 KPI，使外围驱动程序无法轻易访问核心机制。重新实现这些机制很复杂且容易出错 - FreeBSD VM 系统有超过 30K LoC，Linux VM 系统有超过 80K LoC。作为一个真实的例子，Nvidia 的 CUDA 驱动程序实现了完整的 GPU 内存管理系统。实现虚拟地址管理和逻辑映射管理至少需要 11K LoC，物理映射管理需要超过 6K LoC，物理内存管理需要超过 17K LoC。虽然 GPU VM 系统（总共超过 34K LoC）是 Nvidia 的 CUDA 驱动程序（695K LoC）的一小部分 [9]，但它与核心操作系统 VM 系统的大小相当。

Linux帮助设备共享CPU VA空间。 Linux 提供了一些低级别的机制（MMU 通知程序和异构内存管理HMM）来帮助外围设备在共享 CPU 进程的地址空间时与核心 VM 系统进行协调。但是，这些低级别的机制并不能完全消除设备驱动程序实现协调机制的必要性。相比之下，基于 GMEM 的驱动程序利用 GMEM 的高级接口可以无缝共享 VA 空间，因为 GMEM 内部处理了 CPU 和设备之间的协调。此外，Linux 的低级别机制远远不能作为一个外围设备的通用解决方案。仅支持三个核心 VM 事件：销毁映射、限制映射和 CPU 访问设备专用内存。而且，这些事件的协调是“单向”的，即驱动程序可以要求核心 VM 系统在发生这些事件时通知它，但操作系统或其他驱动程序无法要求驱动程序发出类似的通知。

Linux MMU notifier。外围设备驱动程序可以在 CPU 进程的地址空间上创建并插入 MMU 通知程序，并在核心 VM 系统发生某些事件时自动调用驱动程序特定的回调函数，例如，当主机物理地址映射受到限制或被销毁时。驱动程序还可以进一步利用 MMU interval notifier通知程序，该通知程序限制在指定虚拟地址区域内调用回调函数。MMU interval notifier程序有一个内置的间隔树，可以帮助跟踪用户驱动程序的逻辑映射。MMU 通知程序最初是为了简化热拔插访客虚拟机的物理内存的过程而发明的，在这种情况下，访客机必须通过更新访客页表来做出反应。后来，它被 IOMMU 驱动程序用于支持外围设备的共享虚拟内存 (SVM)，其中 CPU 进程的地址空间由支持 IOMMU 的外围设备共享用于 DMA 操作。在 Linux 5.13 中，有九个驱动程序主动使用 MMU 通知程序。其中两个没有实际的外围设备，包括 Xen 和 KVM。其余的用于三个目的 - 使物理地址映射无效，使驱动程序跟踪的逻辑地址映射无效，以及在检测到主机物理地址映射无效时重新提交硬件命令。

异构内存管理。Linux 的异构内存管理 (HMM) 包含几个帮助函数、一个扩展的数据结构和一个通知机制。它们简化了外围设备驱动程序必须执行的协调任务，包括 VA 空间共享和主机-设备内存迁移。要共享 CPU 进程的 VA 空间，外围设备驱动程序会使用以下步骤同步设备的逻辑和物理映射。驱动程序首先使用 HMM 的一个帮助函数来检查 CPU 物理映射。然后，它解析这些映射并建立相同的逻辑映射。根据逻辑映射，可以在设备页表中创建物理映射。此时，设备与 CPU 进程具有相同的内存视图。名为 hmm_range_fault 的帮助函数扫描给定的虚拟地址范围，并可以选择调用 CPU 页面故障处理程序来创建缺失的 CPU 物理映射。Hmm_range_fault 利用 Linux 的 MMU 通知程序来确保在返回之前不会破坏或限制任何主机物理地址映射。设备驱动程序可以使用扩展的数据结构 struct page 和 HMM 的通知机制来实现设备到主机的数据传输。struct page 可用于跟踪设备物理内存的 4KB 粒度，但驱动程序在主机和设备之间迁移内存时必须正确使用 HMM 的三个迁移帮助函数。否则，可能会在核心 VM 系统中引入错误。这正是将核心 VM 的数据结构之一暴露给第三方驱动程序的潜在危险。HMM 的通知机制会在 CPU 进程访问映射到设备本地物理内存的数据时调用由设备驱动程序实现的设备到主机的迁移代码。CPU 页面表被滥用于存储指示页面故障处理程序数据驻留位置的信息。目前，只有 Nouveau 和 IBM 的安全 KVM 在 Linux 5.13 中利用 HMM 的单向通知机制。相反，设备驱动程序必须使用核心 VM 系统中的低级别函数来实现主机到设备的迁移。

和GMEM限制的比较。Linux 的 MMU 通知程序和 HMM 是用于实现外围设备内存管理的低级别机制。它们包括低级别的 KPI 和扩展的内核数据结构，以帮助与核心 VM 系统协调。这种方法有一些缺点。首先，驱动程序必须重新发明自己的 VM 系统，其中大部分可能是硬件独立的。其次，共享低级别的内核数据结构会引入潜在的灾难性错误。如果设备驱动程序没有正确使用低级别的 KPI 来操纵内核数据结构，则整个核心 VM 代码可能会崩溃。这种低级别 KPI 方法最大的缺点是它以一种不希望的方式将设备驱动程序与核心 VM 系统耦合在一起。设备驱动程序可能会无意中误用 KPI，并且核心 VM 系统的所有部分都必须维护这些低级别机制引入的新不变量。这导致了一些微妙的问题。例如，MMU 通知程序的回调函数不允许睡眠，但是驱动程序希望在回调函数中发出设备命令并等待它们。更新核心 VM 系统以允许回调函数休眠将需要设备驱动程序也更新其代码。相反，核心 VM 系统必须理解并正确响应通知事件。例如，Linux 的 out-of-memory reaper 没有调用通知程序，这可能会导致 reaper 已经回收了内存，但设备的驱动程序不知道这一点，从而导致内存损坏。这最终使包括英特尔和 AMD 的 GPU 驱动程序在内的设备驱动程序实现了自己的协调机制，其中 MMU 通知程序只是用作同步机制，以尽量减少核心 VM 系统代码更改的要求。

相比之下，本文提出了 GMEM 来避免上述缺点。GMEM 重构了核心 VM 系统，以提供支持配备各种类型的加速器的机器的集中式内存管理系统。驱动程序开发人员不会看到任何低级别的 KPI、扩展的数据结构或实现指南。相反，设备驱动程序只需要封装它们的硬件相关功能，并将所有硬件无关的功能卸载给 GMEM，使用高级接口。然后，GMEM 会在内部处理协调工作。

GMEM设计和实现

GMEM 以 VA 空间管理为中心，并解耦了 MMU 特定功能。设备驱动程序需要注册自己的 MMU 函数并将设备附加到 VA 空间，如图 1 所示。因此，GMEM 可以内部协调附加设备，以在每个 VA 空间内提供一致的内存视图。因此，设备驱动程序不需要实现任何虚拟内存管理机制。GMEM 的内部协调对于设备驱动程序是透明的。

GMEM 提供了对连接到同一 VA 空间的每个设备对之间的双向协调。这是通过逻辑页表实现的，该逻辑页表可以将虚拟地址映射到任何连接设备的本地物理内存。修改后的页面故障处理程序可以查找逻辑页表以了解物理内存是否已被其中一个连接设备映射。然后，GMEM 决定是为本地内存访问迁移内存还是直接为远程内存访问创建物理映射。两种决定都需要 GMEM 调用两个设备的注册 MMU 函数。这允许（但不要求）驱动程序指示页面故障处理程序在指定的虚拟地址区域内在故障驱动迁移和远程访问之间切换。

设备绑定到VA space

设备驱动程序无需实现任何虚拟内存管理。相反，它们只需要实现 GMEM 所需的 MMU 函数，使用这些 MMU 函数创建 GMEM 设备实例，并将设备实例附加到进程的地址空间。然后，GMEM 会负责设备的 VM 管理，包括调用其 MMU 函数来操作设备页表或使设备 TLB 条目无效。还必须注册额外的硬件信息，包括支持的页面大小、设备物理内存大小、MMU 功能。最重要的 MMU 功能是设备是否可以从硬件页面故障中恢复。如果是这样，设备驱动程序应使用 GMEM KPI 来处理设备页面故障。否则，所有可访问的内存都必须始终映射到设备上。这意味着主机 CPU 必须将附加虚拟地址空间中的所有内存连接起来，以便设备可以访问它。用户可以选择仅连接设备可能会合法使用的内存。这维护了一致性，但限制了设备可以访问的有效虚拟地址。

VA space一致性

提供一致的内存视图可以显著提高用户级可编程性，因为程序员无需分配和管理单独的设备缓冲区。虚拟地址空间一致性依赖于 GMEM 在每个 VA 空间内进行的内部协调。具体来说，GMEM 支持两种协调模式，分别使用共享页表或一致页表。共享页表，顾名思义，实际上是在地址空间的 CPU/设备之间共享的。所有共享者的 MMU 都引用同一个页面表。要使用此模式，所有共享者都必须支持完全相同的页面表格式。但是，它们可以使用不同的 TLB，因此 GMEM 仍需要通过调用每个设备的 MMU 函数来实现 TLB 争夺。

相反，一致页表是一组由 GMEM 保持一致的每个设备页表（使用相同或不同格式）。并非所有一致页表都必须始终包含所有已分配虚拟地址的条目，而是 GMEM 确保设备始终看到一致的内存视图。请注意，逻辑地址映射在一致页表中是相同的，每个页表都可能在任何给定时间安装不同的物理地址映射（对应于那些逻辑地址映射）。

设备可以以不同方式附加到同一个虚拟地址空间。例如，两个设备可以使用共享页表，而第三个设备可以使用一致页表附加到同一个虚拟地址空间。GMEM 将自动保持页面表更新并维护其 TLB 的一致性。

使用一致页表可以灵活地在故障驱动迁移和远程访问之间切换，因为远程内存访问可能会触发第二次页面故障。GMEM 的当前设计将第二次页面故障作为虚假数据局部的提示，并主动将内存迁移到有故障的设备。但是，GMEM 可以进一步扩展，允许程序员定义任意虚拟内存区域的设备亲和性。GMEM 不会迁移内存，而是将其映射到设备的一致页表中的远程物理内存，因此设备会对内存的这些区域进行更慢的远程访问。

GMEM KPI

虚拟地址空间 KPI。GMEM 支持以下虚拟地址空间操作：

  • gm_ret_t gm_as_create(gm_va_t begin, gm_va_t end, gm_as_alloc_t policy, gm_as_t **new_as) 创建一个支持指定虚拟地址范围的虚拟地址空间。虚拟地址分配将根据指定的策略进行，该策略是多种选项的组合，例如首次匹配、保护（在区域的两侧保留一个未使用的页面以捕获越界访问）和缓存（启用已分配但未映射的空闲区域的对象缓存）。

  • gm_ret_t gm_as_destroy(gm_as_t *as) 销毁虚拟地址空间。

  • gm_ret_t gm_as_attach(gm_as_t *as, gm_dev_t *dev, gm_mmu_mode_t mode, bool activate) 使用指定的页面表维护模式（共享或一致）将指定的设备附加到虚拟地址空间。一个设备可以附加到多个虚拟地址空间，但一次只能在一个虚拟地址空间中处于活动状态。如果 activate 为真，则该设备在虚拟地址空间内处于活动状态。

  • gm_ret_t gm_as_alloc(gm_as_t *as, gm_va_t hint, gm_va_t size, gm_va_t align, gm_va_t no_cross, gm_va_t max_va, gm_region_t **new_region) 从虚拟地址空间分配指定大小的虚拟地址范围，并满足给定的约束条件，例如对齐。根据虚拟地址空间的分配策略，给定的提示可能用于指导空闲空间的搜索。分配的区域维护对其包含的地址空间的引用。

  • gm_ret_t gm_as_lookup(gm_as_t *as, gm_va_t addr, gm_region_t **out_region) 在虚拟地址空间中搜索包含指定虚拟地址的已分配区域。

  • gm_ret_t gm_as_synchronize(gm_as_t *as) 等待虚拟地址空间内所有未完成的异步映射更改完成。虽然映射可能对特定设备私有，取决于设备连接期间指定的模式，但所有连接的设备都共享虚拟地址空间内已分配区域的单个一致视图。

设备 KPI。GMEM 支持以下设备操作：

  • gm_ret_t gm_dev_create(gm_mmu_t *mmu, void *mmu_data, gm_dev_cap_t cap, gm_dev_t **new_dev) 创建设备的表示。mmu_data 是一个不透明的指针，它存储在设备表示中并传递给所有 mmu 操作。特别是，此指针与 gm_mmu_CPU 设备一起使用来识别主机 CPU 的虚拟地址空间，该虚拟地址空间的映射将在连接时由 gm_as_t 继承。

  • gm_ret_t gm_dev_destroy(gm_dev_t *dev) 销毁设备。

  • gm_ret_t gm_dev_switch(gm_dev_t *dev, gm_as_t *as) 在指定的虚拟地址空间内激活设备，并隐式地停用其他位置的设备。

  • gm_ret_t gm_dev_detach(gm_dev_t *dev, gm_as_t *as) 将设备从指定的地址空间中分离，并释放任何设备特定的映射结构，例如页表。

  • gm_ret_t gm_dev_fault(gm_dev_t *dev, gm_va_t addr, gm_prot_t access) 当出现地址翻译故障时，由设备驱动程序调用。如果设备能够从地址翻译故障中恢复，并且该函数能够为故障地址建立有效的映射，则该函数的返回值将向驱动程序指示它应该指示设备重试故障访问。

  • gm_ret_t gm_dev_register_physmem(gm_dev_t *dev, gm_pa_t begin, gm_pa_t end) 由设备驱动程序调用来注册设备本地物理内存。一致模式连接的确切行为取决于设备是否声明为能够从地址翻译故障中恢复。具体来说，如果设备无法恢复，则其他设备创建并复制到设备的唯一映射是那些由其创建者显式连接的映射。作为 GMEM 与主机 VM 系统集成的部分，如果主机 CPU 和设备以一致模式附加到虚拟地址空间，则主机 CPU 通过主机 VM 系统接口连接的区域的连接将自动复制到设备。

区域和映射集 KPI。GMEM 支持以下区域和映射集操作：

  • gm_ret_t gm_region_map(gm_region_t *region, gm_prot_t prot, struct vm_page *pages[], gm_mapping_set_t *set, int flags, void (*async_callback)(void *), void *callback_arg) 从区域到指定的物理页面创建一个有线映射。如果 set 不是 NULL，则将映射添加到该集合中。给定的标志可选指定映射的页面大小和/或映射以异步方式创建，在这种情况下，可选的回调将在映射完成后调用。（或者，GMEM 用户可以调用 synchronize 函数来等待完成。）

  • gm_ret_t gm_region_unmap(gm_region_t *region, int flags, void (*async_callback)(void *), void *callback_arg) 取消映射区域。与 map 函数一样，给定的标志可以指定异步取消映射，完成通知通过回调或 synchronize 函数传递。（取消映射区域不会释放区域，它可以由未来的映射操作重新使用。）

  • gm_ret_t gm_region_dealloc(gm_region_t *region) 从其虚拟地址空间中释放该区域。如果区域在释放时仍被映射，则将其取消映射。

  • gm_ret_t gm_mapping_set_unmap(gm_mapping_set_t *set, int flags, void (*async_callback)(void *), void *callback_arg) 行为类似于区域取消映射函数，只是此函数会取消映射整个映射集。完成通知发生在所有映射都被销毁时。

  • gm_ret_t gm_region_set_policy(gm_region_t *region, gm_dev_t *dev, gm_region_placement_t policy) 设置在给定区域出现故障时的位置策略。如果 VA 空间中没有故障地址的物理映射，则 dev 确定在页面故障时应从哪里分配物理内存。具体来说，如果 dev 为 NULL，则将从故障设备的本地物理内存中分配物理内存。如果故障设备没有本地物理内存，则从主机物理内存中分配。每个区域在分配时都有一个默认策略，其中 dev = NULL 且 policy = UNIQUE。UNIQUE 定义 VA 空间中故障地址始终应具有唯一的物理映射。如果是这样（UNIQUE），并且在页面故障时同一 VA 空间内的其他单独页表中存在任何物理映射，则将首先销毁所有这些物理映射。
