物理机下的中断执行


物理设备外部中断(msi)：中断控制器->bus->lapic->cpu

ipi中断：cpu0->lapic0->bus->lapic1->cpu1

外部中断三个必要环节：

  1. 当外部中断到达中断控制器时，我们需要知道中断的目标cpu
  2. 我们需要将中断通过lapic注入到目标cpu上去执行
  3. 目标cpu执行完中断处理程序，需要向中断控制器发送eoi

注：上图中无论handle_irq执行成功与否，apic_eoi都会被执行

以上三个环节是存在cpu-lapic-中断控制器交互的，并不单纯在cpu上执行，也是将场景从物理机迁移到虚拟机上后中断运行需要加速的环节。

ipi中断的执行主要与lapic和cpu相关，流程略有不同。

虚拟机下中断执行中的新角色

vlapic

  • base_address apic-page address
  • regs apic-access page pointer
  • vapic_addr virtual apic page address

Apic-access page可以理解为对物理lapic 寄存器空间的模拟，比如ICR这些寄存器是通过regs+offset访问的，vapic page是对apic page的虚拟化page，结果是减少对apic page访问的vm exit退出，具体之后会介绍。一般情况下，直接访问apic-page，qemu是mmio方式进行的，这会触发ept misconfig导致vm exit。

vmcs & vcpu

vmcs是vcpu上的核心结构，用于配置guest上的预设行为（什么情况下会vm exit，vm-entry可以携带什么参数，打开了什么功能），记录guest的状态（vm exit/entry 上下文保存）。

如果不做任何加速，qemu/kvm下的中断有多惨？

gsi是怎么分配的？

Guest irq是qemu侧分配，通过KVM_SET_GSI_ROUTING传入内核配置

qemu侧的实现里guest irq的状态记录在一个bitmap里

irqfd/eventfd机制

目前，qemu中大量的host-guest 消息通知机制都是通过eventfd进行的，而在中断场景，该eventfd就是irqfd

irqfd机制核心逻辑组成是vfs_poll和eventfd_signal：

  1. 我们通过vfs_poll监听irqfd上是否有信号，这在用户态表现是read（底层会调用vfs_poll），接收到信号后，可以触发对应的绑定操作（schedule_work之类）
  2. 通过触发eventfd_signal向irqfd传递一个信号

但与用户态驱动中的使用略有不同：

  1. 用户态驱动主要是在用户态read以等待内核发上来的信号
  2. qemu/kvm下，在通过kvm_irqfd_assign设置irqfd时，kvm会在内核侧通过vfs_poll自己对创建的irqfd进行监听，如果有虚拟设备/外部中断处理程序通过eventfd_signal向irqfd发中断信号，kvm可直接在内核态调用中断处理程序

最后irqfd机制中，vfs_poll监听到信号，会调用irqfd_inject对中断进行注入，这里会涉及到guest_enter/exit

vm-exit 到 vm-entry

如果在早期我们想要通过host/kvm对guest进行中断注入，那么必然只能在guest enter时注入（通过写vmcs的VM_ENTRY_INTR_INFO_FIELD）。

  1. 如果当前的vcpu已经退出，直接唤醒vcpu即可，直接guest enter
  2. 如果当前的vcpu正在运行，reschedule 该vcpu所在的pcpu，使vcpu短暂让出cpu，这就造成了vm exit，而后vcpu会被快速重新调度running（vm entry/guest enter），这时候VMCS里的内容会被解析成vector，继而给到vcpu执行。

最后这一小节讲一下irqfd_inject（enter_guest之前）是怎么做到的，内核核心实现分为两步：

kvm_make_request

kvm_vcpu_kick

__guest_enter

vcpu_run->vcpu_enter_guest->inject_pending_event->vmx_inject_irq->vmcswrite

IPI中断

IPI中断的路径与外部中断不同，可以理解为vcpu0->vlapic0->bus->vlapic1->vcpu1

ICR寄存器在IPI中断中有着重要作用，ICR与ICR2寄存器中表明着ipi的类型和目标cpu，实际上物理机上IPI通信就是通过写lapic的ICR寄存器接口实现的。

然而到了vcpu上，ICR在哪里？

每个vcpu都有一个对应的vlapic，ICR寄存器的模拟依赖于vlapic->regs + ICR_Offset,也就是通过访问apic-page（mmio region）实现。

大概流程如下：

  1. vcpu对apic-page发起mmio write（ept-misconfig），导致vm-exit
  2. 陷入host，传递到vlapic1，kick vcpu1，vm-entry
  3. Vcpu 1处理IPI请求

中断结束之后（EOI）

除此之外，还有eoi中断，在中断处理完成以后，无论ipi还是外部中断，如果配置了需要eoi，vcpu还会再写一次apic-page的eoi寄存器，还会造成一次vm-exit

中断重映射

iommu有两种中断投递格式，未开启remapping时使用compatible format，即传统投中断，开启remapping后就是我们所说的中断重映射场景，使用remapping格式投递。

中断重映射常用模式两种，remappable和posted，虚拟化主要用posted居多（设备直通）。

Remappable


  • Present域(P)：0b表示此IRTE还没有被分配到任何的中断源，索引到此IRTE的Remapping中断将被blocked掉，1b表示此IRTE是有效的，已经被分配到某个设备。
  • Destination Mode域(DM)：0b表示Destination ID域为Physical APIC-ID，1b表示Destination ID域是Logical APIC-ID。
  • IRTE Mode域(IM)：0b表示此中断请求是一个Remapped Interrupt中断请求，1b表示此中断请求是一个Posted Interrupt中断请求。
  • Vector域(V)：共8个Byte表示了此Remapped Interrupt中断请求的vector号(Remapped Interrupt)。
  • Destination ID域(DST)：表示此中断请求的目标CPU，根据当前Host中断方式不同具有不同的格式。xAPIC Mode (Cluster)为bit[40:47]， xAPIC Mode (Flat)和xAPIC Mode (Physical)为bit[47:40]， x2APIC Mode (Cluster)和x2APIC Mode (Physical)为bit[31:0]。
  • SID, SQ, SVT则联合起来表示了中断请求的设备PCI/PCI-e request-id信息。

Posted

所有的Remapping格式中断请求都需要通过中断重映射表来投递， IRTE中的Mode域(IM)用来指定这个remappable中断请求是interrupt-remapping方式还是interrupt-posting方式。

  • IRTE的IM位为0表示中断按照remappable方式处理；
  • IRTE的IM位为1表示中断按照posted方式来处理。

IRTE的格式相对于Remapping模式新增了以下几个域：

  • 中断请求对应的Posted Interrupt Descriptor数据结构地址，包含高地址和低地址2个域；
  • Urgent (URG)标志来指定中断请求是否需要实时处理；
  • 一个用来指定要post的vector号的Vector域，与Remapping格式不同的是posted-format的IRTEs的Vector域是用来决定Posted Interrupt Descriptor里的PIR域的哪个bit要置位。

Vt-d post interrupt

descriptor


  • Posted Interrupt Request (PIR)域，提供记录需要post的中断占256bit每个bit代表一个中断号。
  • Outstanding Notification (ON)域，由硬件来自动更新，用来表示是否有中断请求pending。当此位为0时，硬件通过修改其为1来产生一个通知事件告知中断请求到来。接收这个通知事件的实体(处理器或者软件)在处理这个posted interrupt后必须将其清零。
  • Suppress Notification (SN)域，表示non-urgent中断请求的通知事件是否要被supressed(抑制)。
  • Notification Vector (NV)域，用来指定产生posted-interrupt“通知事件”(notification event)的vector号。
  • Notification Destination (NDST)域，用来指定此中断要投递的vCPU所运行物理CPU的APIC-ID。

其他支持

  1. VMM需要在每个PCPU上安排2个中断vector用来接受通知事件：
  • 其中一个物理vector被称之为Active Notification Vector (ANV)，它被用来post通知事件到处于Running状态的VCPU上（这个IPI中断是guest接收的）。
  • 另一个物理vector被称之为Wake-up Notification Vector (WNV)，它被用来post通知事件到处于Blocked状态的VCPU上（这个IPI中断是host接收的）。
  1. 需要VMCS支持virtual apic page
  2. 需要VCPu支持apicv特性
  3. iommu使能posted remmaping

调度

在VCPU调度的过程中，VMM需要按照下面的方式来管理VCPU的调度状态：

  • 当VCPU被scheduler选中调度进来运行的的时候，此时VCPU的状态被标志为'Active'状态。这个时候VMM需要将PD的NV域更新为ANV的值。 同时在这种场景下，此VCPU上接受的Posted Interrupt中断会被直接复制到vAPIC Page中，guest在非根模式下就能直接处理此中断，而不需要VMM的参与。
  • 当一个VCPU被抢占（Preempted），例如时间片到期了，这时候需要将PD的SN域标志为1，即将VCPU更新为'Preempted'状态，告诉硬件当前VCPU已经没在非根模式下运行了。 此时，这个VCPU上的non-urgent中断都会被接受但不会产生通知事件。 但如果这个VCPU上有标志为URG类型的中断时，VMM同时也会将PD的NV域修改为WNV，这样一来VMM就能够将URG中断请求投递给处于not running状态的VCPU，并进行适当的软件处理（例如，抢占正在同一个物理CPU上运行状态的其他VCPU，并将自己调度进来）。
  • 当一个VCPU执行了hlt指令或者触发了ple，VMM也会干预进来将VCPU给block出来，并且将VCPU状态标识为Hlted状态。在此状态下VMM需要将VCPU对应的PD的NV域设置为WNV。这样一来，当中断请求post到此VCPU时，VMM能够接收到Wake-up Notification Event事件通知并做出适当的软件操作。（例如：立即对此VCPU进行一次调度）

当VCPU重新进入非根模式或者从hlt恢复执行时（注意这个时候vCPU还没进入根模式下），VMM对此VCPU上处于pending状态的posted interrupt进行处理。

当VCPU在不同PCPU之间进行迁移时，VMM会对VCPU对应的PD的NDST域进行更新，将NDST改为VCPU要迁移的目的PCPU的APIC-ID。这也就是说，在VCPU迁移的过程中我们也顺便完成了中断迁移。这样一来，新的posted interrupt 通知事件到来时就会被自动路由的新的PCPU上

软件code的执行？

post-interrupt下，软件的实现主要是配置工作，也就是填irte表和写post-interrupt descriptor，配置完成后，对中断实际投递和处理参与很少（主要是wnv和anv两个中断code）。

Virtual apic-page access & apicv特性

通过posted-interrupt我们几乎解决了之前的问题，但还有这么些问题需要回答：

  • eoi是怎么处理的？
  • IPI下怎么vm-exit优化的？
  • posted-interrupt通过vapic-page写vector注入中断，为什么vapic-page 可以？

vapic提供了如下主要能力：

  1. virtual-interrupt delivery: VMM直接写Virtual-APIC Page的VIRR寄存器就能给vCPU投递中断。
  2. virtualize APIC accessess：vCPU可以通过MMIO方式访问APIC寄存器；
  3. virtualize x2APIC mode：使能基于MSR方式对APIC寄存器的访问（x2API虚拟化）；
  4. APIC register virtualization： 控制APIC寄存器的访问方式（MMIO/MSR BASED）将MMIO写APIC-access page操作重定向到virtual-APIC page上。
  5. Process posted interrupts：VT-x Posted Interrupt 特性，配置Posted Interrupt Descriptor Adress和 Notification Vector， 当目标处理器接到中断后，硬件自动将中断请求信息copy到virtual-APIC page上。

基本原理：guest与host协商一块内存用于通信，绕开vm-exit的路径。
