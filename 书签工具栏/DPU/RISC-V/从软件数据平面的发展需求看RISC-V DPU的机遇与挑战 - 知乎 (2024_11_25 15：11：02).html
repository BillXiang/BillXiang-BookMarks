一、前言

软件定义网络(Software Defined Network)和网络功能虚拟化技术(Network Function Virtualization)的提出带来了网络领域的革命性变化。其中，SDN注重于控制平面与数据平面的分离，控制平面负责流量和路由策略的控制，数据平面负责数据包的转发交换，网络的可编程化配置使其更具灵活性。而NFV则更注重于软件与硬件的分离，利用虚拟化技术摆脱专用硬件的限制，显著降低了运维成本。上述两种技术相互独立，却又相辅相成，共同引导并推进云服务、数据中心等网络基础设施的发展。

数据平面开发套件(Data Plane Development Kit)为软件数据平面(Software Data Plane)的搭建提供了丰富的库函数和驱动支持，在用户空间实现了数据包的高性能处理。DPDK还为虚拟化网络功能(Virtual Network Function)提供了Virtio、Vhost、vDPA等技术支持，是许多NFV应用程序的核心。近期，赛昉科技也在社区开源了DPDK在RISC-V平台的移植部署工作，这为我们带来了许多新的思考。

二、数据平面开发套件(DPDK)的原理简介

在正式介绍DPDK之前，不妨先回忆一下C10M问题。在当时，工程师们已经利用事件驱动(epoll)的方式解决了C10K问题，但接踵而来的困难是，如何将服务器所能支持的并发连接数量scaling至千万条？

Robert Graham在Shmoocon 2013 - C10M Defending The Internet At Scale的讲座中指出，该问题的关键在于内核的可扩展性！显然内核是没办法scaling的，那么最佳的解决方案就是直接跳过内核，自己处理所有相关的事务。

DPDK就是利用UIO技术将设备的物理空间映射到用户的进程空间，在用户态实现设备的读写操作以及中断处理，从而绕过了Linux内核态的网络协议栈。

在传统的基于Linux内核的数据包处理模式下，网卡在接收到数据包后，首先会根据接收描述符队列的头指针找到空闲的描述符，用DMA将数据包传输至描述符指向的环形缓冲区中，然后修改描述符状态并产生硬中断通知CPU。此时CPU在接收到中断后，会从用户态切换至内核态，并在响应完中断处理函数后，产生软中断。在软中断处理函数中，内核借助网络协议栈进行相应的数据包处理，然后将用户所需的数据从内核空间拷贝至用户空间，并最终再从内核态重新切换回用户态。故综合来看，在传统的非NAPI模式下，单个数据包的收发处理会带来两次中断、两次上下文切换以及两次数据拷贝的巨额开销。

DPDK则在UIO驱动中屏蔽了设备中断，采用轮询(poll)的方式来实现数据包的收发，在IO密集型应用场景下，可以防止CPU被频繁中断，减少了软硬件中断带来的开销。网卡DMA将数据包拷贝到位于内存池的mbuf数据结构体中后，用户可以直接获取数据包，实现所谓的"零拷贝"操作。最为关键的是，DPDK在用户态完成了数据包的处理过程，减少了上下文切换，显著提升了数据包的收发性能。

三、DPDK的性能优化技术

为进一步提升性能，DPDK从数据结构、软件算法、线程等角度出发均进行了深度优化。在数据结构方面，DPDK提出mbuf数据结构封装控制信息和数据包以提高访问效率，采用无锁队列节省竞争锁的开销，使用图结构作为数据转发的框架， 将处理逻辑抽象成节点以提高复用性和灵活性等。在软件算法方面，DPDK实现了桶搜索算法、分层调度等算法的流水化处理，对路由查表算法(DIR-24-8)、过滤器查找、Hash表遍历等进行了加速优化。在线程方面，DPDK利用CPU核的亲和性，将线程与逻辑核绑定，提高缓存的命中率，减小内存访问损耗。

DPDK采用了许多软件优化技术，还包括缓存行对齐、减少数据伪共享、使用本地局部变量等等，这里不再一一进行罗列。但作为硬件设计工作者，我们不仅需要了解应用的行为特点和技术优势，还需要关注如何通过硬件架构设计来提高软硬件协同加速的能力，所以下面详细介绍一下DPDK所采用的硬件优化技术。

3.1 SIMD技术

SIMD(Single Instruction Multiple Data)，即单条指令完成多个数据的向量化处理，旨在提高数据处理的并行度。X86架构下的AVX(Advanced Vector Extensions)、SSE(Streaming SIMD Extensions)扩展指令集，ARM架构下的NEON指令集都属于最典型的SIMD指令。

为支持SIMD指令扩展技术，处理器需要扩展高数据位宽的向量寄存器、向量计算功能单元以及相应的向量载入/存储单元，以实现数据的并行化计算处理。在该类指令架构的支持下，DPDK可以实现数据包的批量并行处理，提高存储带宽的利用率，在最长前缀匹配(LPM)、访问控制列表(ACL)等库函数以及三层转发l3fwd等示例程序中也都大量运用了向量指令优化技术。当然最具代表性的，莫过于rte_memcpy()内存拷贝优化函数，利用SIMD技术可以提高数据搬移的效率，最大化使用访存带宽，还能弥补数据非对齐所带来的部分性能损失。

3.2 缓存预取技术

利用数据在时间和空间上存在局部性的特点，缓存预取技术可以在处理器显示访问前，将相邻的若干个缓存行数据提前获取至缓存中，减少缓存缺失率，从而隐藏内存访问延时，实现处理器性能的进一步提升。DPDK大量使用软件预取指令进行性能优化，例如mbuf数据结构中元数据字段的预取用于加速数据包处理和释放，Hash散列表中键值的预取用于加速多级表项的映射查找等等。

X86架构支持的预取指令包括prefetcht0、prefetcht1、prefetcht2和prefetchnta，ARM架构则支持PRFM PLDL1KEEP、PRFM PLDL2KEEP、PRFM PLDL3KEEP和PRFM PLDL1STRM，这两种架构均支持非临时预取，且可以针对不同的缓存层级，实现针对性的软件预取操作。

3.3 巨页技术

标准的页大小为4KB，其记录有虚拟地址和物理地址的转换关系，内存管理单元(MMU)则负责借助TLB和页表查询来实现地址的翻译功能。传统的处理器架构只需要进行一次地址转换，而内存虚拟化技术要求实现GVA(Guest Virtual Address)到GPA(Guest Physical Address)再到HPA(Host Physical Address)的两次地址转换，这使得页表层级增加，访问遍历时延增长，也使得TLB的缺失代价变得更加高昂。为此，DPDK选择采用2MB和1GB的巨页技术(HugePage)，通过增加页大小来减少页表需要遍历的级数，并降低TLB的缺失率。

目前MMU普遍支持4KB、2MB和1GB大小的页管理，X86架构VT-d技术的IOTLB支持2MB和1GB大小。因此为提高DPDK的性能，需要在设计实现MMU和IOMMU单元时，额外提供上述两种标准规格的巨页技术支持。

3.4 DDIO技术

DPDK提供了对Intel X86架构特有的DDIO(Data Direct I/O)技术支持，网卡等IO设备可以直接与最后一级缓存(LLC)进行数据交互，从而减少内存等慢存储设备的访问，提高数据包收发的效率。以一个理想的数据包接收过程为例，在不支持DDIO的情况下，网卡需要把数据包填充至内存中的数据环形缓冲区内，等待处理器进行访问并获取至各级缓存中，而在DDIO技术的加持下，数据包被直接存放在LLC中，处理器无需访问内存即可获得待处理的数据，可以显著减少数据包的处理延时。

由于LLC除正常的缓存数据外，还需要额外存储IO设备的数据，因此必须对其进行适当的扩容，增加片内的存储面积并添加相应的缓存控制逻辑。以英特尔的E5处理器系列产品为例，其LLC Cache的容量提高到了20MB。

四、DPDK——DPU的前世今生

无论是IO半虚拟化还是IO透传，DPDK都提供了良好的技术支持。针对半虚拟化技术，其在用户态分别实现了Virtio-net前端驱动和Vhost-user后端驱动。针对IO透传技术，DPDK实现了vDPA(vhost Data Path Acceleration)，利用Vhost-user屏蔽底层硬件驱动的具体实现，使Virtio-net驱动能与vDPA设备直接进行数据平面的交互，有效解决SR-IOV(Single Root I/O Virtualization)的热不可迁移问题，同时降低了网卡对virtio接口的支持难度。

考虑到DPDK在数据平面的高性能处理能力以及对IO虚拟化技术的良好支持，许多VNF选择基于DPDK进行应用加速，其中最典型例子的就是OVS-DPDK。

Open vSwitch是一类开源的虚拟交换机，承担着云计算中网络虚拟化的主要工作，主要负责同主机、不同主机间的虚拟机通讯，以及虚拟机与外部的网络通讯。OVS在内核态用openvswitch.ko实现Fastpath，在用户态用ovs-vswitchd实现Slowpath，而OVS-DPDK可以通过PMD轮询驱动直接将数据包送至用户态，实现内核的旁路功能，据测试DPDK可以为OVS带来接近十倍的性能提升效果。

如果说NFV是一种新型的网络设备部署方式，那么VNF就是其中某个被实例化的虚拟网络设备或节点，而DPDK对于VNF性能加速的重要性，就直接决定了其在网络功能虚拟化技术中具有举足轻重的地位。同时，DPDK作为SDN中的数据平面，切实将两者的理念结合了起来，因此日益受到业界的关注与研究。

作为DPDK的姊妹，SPDK(Storage Performance Development Kit)同样利用轮询和内核旁路的方法，来优化存储IO性能，DPDK与SPDK已俨然成为SDN和SDS(Software Defined Storage)的典型代表。目前，DPDK库函数除了基本的以太网设备(ethdev)外，还支持了加密设备(cryptodev)和压缩设备(compressdev)。当这些熟悉的名词交织在一起时，不禁让我们联想到近期非常火热的一个概念——DPU！

谈及DPU，可以先从Amazon的"Datacenter Tax"说起。即在数据中心搭建过程中，为实现网络、安全等基础设施服务，就需要消耗相当一部分算力，这些硬件资源的消耗就像税一样附加在用户购买的云服务当中。随着IO数据的与日俱增，CPU显然开始不堪重负，这部分“CPU处理性能低下、GPU处理不了”的IO密集型任务需要从CPU中进行卸载(Offload)，而DPU就承担这部分使命工作，成为继CPU、GPU后的“第三颗主力芯片”。

那么作为XPU的新成员，DPU具体指代什么呢？目前业内也仍未给出统一的答案，其中主流的说法有以下三种。数据处理单元(Data Processing Unit)：区别于CPU的控制功能以及GPU的图像处理功能，DPU主要负责为数据处理提供加速，而这些"数据"主要指网络数据包、数据库的结构化数据表等数字化信息，这里强调了DPU的功能以及面向的对象。数据中心处理单元(Datacenter Processing Unit)：对于现代大型数据中心而言，计算、网络和存储是最为关键的基础服务设施，通过让CPU负责控制、GPU负责辅助计算、DPU负责网络和存储管理，三者各司其职，可以用更低的成本实现更高的性能，而这里阐明了DPU的一种特定应用场景。以数据为中心的处理单元(Data-centric Processing Unit)：不同于以控制为中心(Control-centric)的设计观念，DPU将数据的流式化处理(Flow)作为设计核心，打破传统冯诺依曼架构中存算分离的局限性，实现更高效的计算处理，这里则突出了DPU的设计理念。

显然，上述三种解释并不是相互对立的，它们只是从不同的角度对DPU的基本特征进行了描述，即DPU的核心在于"数据"，它需要帮助CPU实现网络虚拟化、硬件资源池化等基础服务功能的卸载，为网络、存储和安全等数据处理提供性能加速，并最终达到降本增效的目的。

DPDK其实已经具备了DPU所需的部分功能，换句话说，DPDK可以直接部署在DPU的可编程软件框架下，提供部分基础服务。在2020年的GTC大会上，英伟达推出了BlueField-2 DPU(Data Processing Unit)，并介绍了基于DPDK、SPDK、P4等开放API的DOCA(Data-Center-Infrastructure-On-A-Chip Architecture)软件生态架构，目前官网也已经发布了许多关于DPDK在BlueField DPU中的应用教程，例如《使用 NVIDIA BlueField DPU 和 DPDK 开发应用程序》和《在 BlueField DPU 上实现OVS DPDK》等。而英特尔推出的IPU(Infrastructure Processing Unit)，同样在其官方的产品介绍中说明 "开源软件设计：基础架构程序员开发套件 (IPDK) 利用并扩展了数据平面开发套件 (DPDK) 和存储性能开发套件 (SPDK)"。

目前，芯片市场上还未出现基于RISC-V架构的DPU产品，或许我们可以先从以DPDK为代表的软件数据平面出发，根据其发展需求得到部分技术思考，用于指导基于RISC-V指令集架构的DPU设计工作。

五、RISC-V DPU的机遇与挑战

机遇：

1. 开源的指令级架构

RISC-V指令集架构最大的优势在于开源开放。众所周知，ARM指令集架构的许可证是非常高昂的，且在授权到期后，需要就继续授权和授权费用问题重新进行谈判。而对于RISC-V指令集架构以及相关的开源软件，企业和用户可以自由使用和开发，还可以将自主研发的产品进行二次开源或者商业收费，这对于小型团队和商业公司来说都非常友好。

2. 指令集的模块化拓展

RISC-V指令集与微架构的松耦合性，提高了架构设计的自由度，硬件架构师们可以按需进行指令集的模块化扩展和专用领域架构的灵活设计。RISC-V的向量指令集扩展(V)在去年九月份定稿，基础缓存管理操作指令扩展(CMO)以及密码学指令集扩展(Crypto)，也于今年相继定稿。其中，向量指令和缓存预取指令的拓展可以提高RISC-V处理器的网络数据处理能力，密码学指令的拓展可以提高其数据加解密和提供网络安全服务的能力，而用户自定义的指令亦可以在特定应用场景下实现性能的提升。

3. 硬件敏捷开发

传统的硬件描述语言如Verilog等抽象层级低，代码的复用性差，致使硬件开发效率低下，迭代周期较长。而以Chisel为代表的新型硬件描述语言，具有更高的抽象层次，显著提高了代码的复用性，可以实现数字芯片的快速开发与迭代。借助芯片敏捷开发工具，用户可以快速实现DPU中处理器芯片和专用加速处理器等硬件单元的设计工作，提高开发效率，节省研发成本。除此之外，借助Chisel的参数化能力还可以实现设计空间的快速探索，得到最优的硬件配置参数。

4. 更高的能效比

利用RISC-V指令集精简规则、架构设计自由灵活的优势，用户可以按需裁剪并增添专用指令，使其以更小的内核面积，更低的功耗，实现高性能表现，显著提高能效比。目前，基于RISC-V指令集架构的超高性能处理器也相继问世，例如阿里的玄铁C910处理器、SiFive的P650处理器等等，在保证面积和功耗优势的前提下，进一步提高了性能极限。数据中心是典型的高耗能产业，服务器需要24小时不间断运行，电费占据了接近80%的运行成本，因此功耗和性能的转换效率比至关重要。而RISC-V DPU可以借助架构优势，将能效比作为核心优化方向，在保证高性能的同时减少面积功耗，节省运维成本，从而实现弯道超车。

挑战：

1. 扩展指令集的实现与自定义指令

实现RISC-V官方的指令集拓展，可以有效提高RISC-V处理器的网络数据处理、安全加密等能力，但同时增加了处理器的功耗与面积。这样的设计思路其实依旧以CPU作为运算核心，并没有摆脱传统的设计观念，很难实现任务的有效卸载。在架构探索过程中，可以仅将CPU作为控制的核心，按应用需要为RISC-V处理器进行指令集的选择性拓展，将部分功能交由专用的加速单元处理。

为实现RISC-V DPU对特定应用场景的高效加速，架构设计者可以选择实现自定义指令，也可以通过ASIC专用加速引擎完成，亦或是使用可编程逻辑进行可重构加速，实现通用处理器与专用加速器之间的性能与效率的权衡。

2. 存储结构的优化

相对于传统的CPU，DPU的缓存结构有比较大的改变。在IO密集型应用场景下，数据的时间局部性受到淡化，以空间局部性为主，例如单个数据包仅会被处理一次。但是存储单元的访问次数却更加密集了，致使缓存出现大量的数据缺失，成为系统的性能瓶颈之一。DPU进行数据处理时，要求数据在其中的停留时间足够得短，才能达到满载吞吐量。这种以数据流为主的流式处理方式，流程上与GPU相类似，多级缓存的功能被显著弱化，所以需要精简缓存结构并提高缓存容量。利用缓存预取、DDIO等技术可以减少缓存缺失，使用紧耦合存储TCM(Tightly Coupled Memory)可以达到更低的延时要求，进一步优化数据通路。

DDIO：DDIO技术在本文的3.4小节已经有过简单的介绍，该技术由英特尔公司提出，其目的是为了让服务器能够更快速地处理IO数据，提高系统吞吐率。对于单个数据报文的收发处理，CPU和网卡需要多次访问内存，而DDIO可以使两者通过LLC直接进行数据交换，绕过了内存这个相对慢速的部件。由此降低了报文的处理延迟，减少了能耗，但该技术需要增加缓存容量并修改相关的控制逻辑。

TCM：TCM是一块固定大小的RAM，它紧密地耦合至处理器内核，发挥着类似于缓存的作用，其访问延时仅为1个时钟周期，远端TCM的访问可能会产生额外的花费。TCM中的内容是通过处理器或是硬件调度器将数据或函数放置在固定位置，以获得较低的延时，其地址范围在系统生成时确定，与通过系统互连结构连接的其他内存设备相同。用户可通过程序来控制TCM，因而其比统计复用的缓存有更好的可预测性，通过将加密算法、编解码代码存在ITCM中，可以降低访存延时，提高系统性能，但也可能带来新的数据一致性问题。

3. 异构多芯粒的高速数据一致性互连问题

DPU是异构计算的阶段性标志，其充分结合利用CPU、GPU、FPGA等异构资源的各自优势进行合理分工，以提供更高性能的基础设施服务。为了实现数据的高并发处理，DPU还需要采用多核或众核结构，集成足够数量的核心来满足性能需求，并要求具备良好的可扩展性。这种新颖的异构众核结构，既需要保证数据处理与交互的灵活性，又需要提供极高的数据传输带宽，无疑提高了数据一致性互连方案的解决难度。

芯粒(Chiplet)或许是异构集成芯片的有效解决方案，通过将原本复杂的SoC系统按照功能和计算单元进行分解，Chiplet可以实现芯片级的IP复用，利用先进2.5D/3D封装技术将模块化的裸片进行互连，从而降低设计复杂度与研发成本。但在此时，传统的片内一致性互连协议无法再保证片间数据传输的速度、延迟和功耗，众多芯片厂商都纷纷推出了自己的互连协议标准，例如英伟达的高速互联NVLink方案，英特尔的AIB高级接口总线协议，AMD的Infinity Fabric总线互联技术等等。上述的这些高速互联协议标准虽然能很好地解决部分性能问题，但是都大相径庭，无法满足不同厂商间的芯粒互连需求，限制了Chiplet技术的"万能胶水"功能。

UCIe(Universal Chiplet Interconnect Express)标准的提出，真正实现了Chiplet接口的标准化，进一步推动了芯片级异构计算的发展。UCIe由英特尔与AMD、ARM等行业巨头共同打造，它规定了芯粒间互连的开放式标准，目前主要兼容PCIe(Peripheral Component Interconnect Express)、CXL(Compute Express Link)等多种互连协议。基于UCIe标准，用户可以在异构集成芯片的设计过程中，选择不同厂商、不同工艺下的Chiplet小芯片来实现混合配置，打破以往的技术壁垒。除了UCIe标准外，CCIX(Cache Coherent Interconnect for Accelerators)协议也是一个在构建异构计算系统时颇具代表性的缓存一致性互连标准，它在继承PCIe兼容性的基础上，有效解决了传统PCIe总线缺乏数据一致性机制的问题。通过自动保持主设备和加速器间的数据一致，CCIX无需依靠驱动实现数据搬移，降低了延时和软件开销，同时提升了易用性。

无论是异构SoC还是异构Chiplet，其一致性互连协议都需要面对不同芯粒/XPU数据流迥异这一严峻挑战。简单的同构总线协议可以简化软件编程复杂度，实现系统资源的协调互助，但难以满足异构系统下多样化的数据流访存需求，部分加速器的数据流在适用于CPU的总线协议下存在显著性能瓶颈。而不同类型的计算单元采用不同的总线协议需要相应的配套代码，多种总线协议的共存需要额外的编程范式规定跨总线的数据交互，增加了软件编程复杂度，且不同总线上的资源在系统负载不均衡时存在资源利用率较低的情况。如何实现一种能够兼容不同数据流类型的高速一致性互连协议，而不是简单地做一些协议转换，这仍是工业界和学术界亟待攻克的难题之一。

4. IO虚拟化技术支持不完备

IOMMU(Input/Output Memory Management Unit)，即输入输出的内存管理单元，主要提供DMA(Direct Memory Access)和中断的重映射功能。我们知道核内的MMU单元可以实现不同进程间物理地址的隔离，支持虚拟地址与物理地址的转换和访问权限的检查，IOMMU具备与其相似的能力，只是其针对的对象是IO设备。IOMMU的出现，使得DMA使用虚拟地址进行数据传输，防止其利用物理地址非法破坏重要的数据区域，对系统的安全构成威胁。不仅如此，它还使得宿主机的物理地址不会直接暴露给虚拟机，为IO透传提供保护，从而提高虚拟化网络的传输性能。

X86架构的VT-d技术，ARM架构的SMMU单元都具备上述功能，但RISC-V架构的IOMMU Spec目前还正在草案拟定当中，部分技术实现可以参考阿里玄铁IOMMU的设计方案。

5. 数据中心工作负载的多样性

数据中心的工作负载类型复杂多样，用户也存在众多个性化的需求，DPU很难包络万象，将所有的基础服务进行卸载加速。专用ASIC硬件加速引擎固然是最高效的，但通用性较差，无法满足多元化要求。使用通用处理单元、可编程逻辑单元或额外增添专用加速模块都可以提高DPU的通用性，但也增加了面积与研发成本，影响实际的能效表现。因此需要进行合理的市场定位，兼顾效率与通用性的设计。

6. 软硬件协同设计

异构计算缺乏通用性，给软硬件开发带来了更高的复杂度，因此需要将软件硬件作为一个整体来设计，提高DPU的易用性。英伟达的DPU部署了通用ARM核用于系统编程，并提供DOCA编程框架提高开发效率，而英特尔的IPU使用FPGA实现可编程功能，并提供IPDK编程框架。同样，未来可能出现的RISC-V DPU也需要提供对于开发人员友好的通用编程框架，构建良好的软件开发生态，提高设计、维护和产品迭代的效率。
